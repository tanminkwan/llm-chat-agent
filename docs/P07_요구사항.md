# Phase 07 요구사항 정의서 (Tool Calling 시뮬레이터 — LLM 추론·응용력 평가)

## 1. 개요
LLM 의 **의미 분석(semantic understanding)** 및 **추론(reasoning)** 능력은 단순 벤치마크 점수만으로는 가늠하기 어렵다. 운영 환경에서 더 직접적으로 드러나는 지표는 **"같은 도구를 손에 쥐어줬을 때 얼마나 멀리까지 워크플로를 스스로 조립해 내는가"** 이다.

- **추론 능력이 좋은 LLM**: CRUD 수준의 원자적(atomic) 도구만 줘도 자연어 요구를 분해(plan) → 도구 다회 호출(multi-step tool calling) → 결과 합성으로 워크플로를 스스로 만든다.
- **추론 능력이 약한 LLM (작은 모델 등)**: 같은 원자 도구로는 워크플로를 못 짜며, **사람이 미리 조립해 둔 매크로 도구(=high-level workflow)** 를 도구 카탈로그에 넣어줘야만 동작한다.

본 Phase 는 이 가설을 **현장에서 직접 비교·실증** 할 수 있는 **Tool Calling Simulator UI** 를 구축한다. 운영자는 UI 에서

1. **JSON 스키마(이름·description·parameters·returns)** 와 **Python 본문 코드** 를 등록하면, 서버가 이를 **동적으로 함수로 컴파일·등록** 하고
2. **자연어 프롬프트** 를 입력하면, 등록된 도구 카탈로그를 **LLM 에게 노출** 하여
3. LLM 이 **어떤 도구를, 어떤 인자로, 몇 번 호출했는지** 와 **최종 응답** 을 한 화면에서 확인한다.

같은 자연어 프롬프트를 두고 (a) 원자 도구만 등록한 카탈로그 vs (b) 매크로 도구를 추가한 카탈로그 / 또는 (a') Reasoning LLM vs (a'') Chat LLM 을 **A/B 비교** 함으로써, 모델별 응용력을 정성·정량 평가한다.

본 Phase 의 산출물은 **운영 트래픽과 분리된 독립 평가 도구** 이며, 기존 `/chat`, `/api/rag/*`, `/api/prompts/*` 의 동작에는 영향을 주지 않는다.

## 2. 목표
- **동적 도구 정의**: UI 에서 JSON 메타데이터 + Python 본문을 입력하면 서버가 즉시 LangChain Tool 로 등록·교체·삭제할 수 있다.
- **자연어 → Tool Call 시각화**: 자연어 입력 한 번에 대해 LLM 이 발생시킨 **모든 tool call 시퀀스(이름, 인자, 반환값, 소요시간, 토큰 사용량, 최종 응답)** 를 단일 화면에 트레이스로 노출한다.
- **모델 비교 가능**: 동일한 도구 카탈로그·동일한 자연어 입력에 대해 `CHAT_LLM` / `REASONING_LLM` 을 토글하여 결과 차이를 즉석 비교할 수 있다.
- **카탈로그 비교 가능**: 도구를 "활성/비활성" 토글하여, **CRUD 원자 도구 세트 vs 매크로 도구 추가 세트** 를 즉석으로 갈아끼우며 동일 LLM 의 응용력 차이를 실증할 수 있다.
- **격리된 실행 환경**: 동적으로 등록된 사용자 코드는 **샌드박스(허용 모듈 화이트리스트 + 실행시간/메모리 제한)** 안에서만 실행되어 운영 컨테이너에 영향을 주지 않는다.
- **재현성**: 등록한 도구 정의·자연어 입력·실행 트레이스는 DB 에 저장되어, 후속 비교·회귀 검증에 재사용 가능하다.

## 3. 범위 (In / Out of Scope)

### 3.1 In Scope
- 새 메뉴 **"🛠 Tool Lab"** 추가 — 사이드바 진입점 + 전용 페이지(`/toollab`).
- 도구 정의 CRUD API 및 UI (목록/생성/수정/삭제/활성·비활성 토글).
- 도구 동적 컴파일·등록 (애플리케이션 메모리 + DB 영속).
- 자연어 프롬프트 → tool calling 실행 → 트레이스 표시 API/UI.
- 단일 모델 토글(`CHAT` ↔ `REASONING`) 실행.
- 트레이스 영속화(요청·tool call 단위) 및 LLM_LOG 적재(Phase 06 와 동일 포맷).
- 샘플 도구 시드(원자 4종 + 매크로 1종)로 첫 기동 후 즉시 데모 가능.

### 3.2 Out of Scope (본 Phase 미포함)
- 멀티모델 동시 실행/벤치마크 자동 채점, 점수표 산출(향후 Phase).
- 외부 네트워크/디스크에 접근하는 도구(파일 I/O, 외부 HTTP 호출 등) — 보안상 본 Phase 에서는 불허.
- 다른 사용자의 도구 공유/팀 워크스페이스(권한 분리는 단순 Admin/User 수준만).
- LangGraph·MCP 등 외부 에이전트 프레임워크 통합.

## 4. 시스템 구성

### 4.1 아키텍처 변경 요약
- **신규 컨테이너 없음.** 기존 `llm-agent` 컨테이너 안에 라우터·서비스·정적 페이지만 추가한다.
- **신규 의존성**: 별도 외부 패키지 추가 없음. 기존 `langchain-openai` 의 `bind_tools()` 기반 OpenAI 함수콜 규약을 사용한다. 샌드박스는 표준 라이브러리 `RestrictedPython` 등 외부 의존을 도입하지 않고, **AST 화이트리스트 검사 + `exec` 격리 네임스페이스 + 별도 프로세스/스레드 타임아웃** 으로 구현한다(설계서에서 상세).
- **DB 변경**: 신규 테이블 2개(`tool_definitions`, `tool_runs`).
- **UI 변경**: `index.html` 사이드바에 진입점 추가, 신규 `toollab.html` + `toollab.js` 추가.

### 4.2 데이터 흐름
```
[Tool Lab UI] ── 도구 등록(JSON+code) ──► /api/toollab/tools (POST/PUT) ──► AST 검증 ─► 컴파일 ─► 메모리 레지스트리 + DB
[Tool Lab UI] ── 자연어 입력 ─────────────► /api/toollab/run ──► LLM(bind_tools) ─► (tool_call 루프) ─► 트레이스 응답
                                                                              │
                                                                              └──► tool_runs 테이블 + LLM_LOG (Loki)
```

### 4.3 LLM 백엔드 호환성 매트릭스 (Tool Calling)
본 Phase 는 LLM 호출을 LangChain `ChatOpenAI` (OpenAI-compatible) 단일 클라이언트로 통일하며, provider 별 응답 포맷 차이는 `bind_tools()` 가 흡수한다. 즉 **클라이언트 코드는 백엔드별로 분기되지 않는다.** 단, vLLM 백엔드는 **모델별 `--tool-call-parser`** 가 정확히 켜져 있어야 OpenAI-compatible `tool_calls` 가 구조화 응답으로 돌아온다. 미설정 시 모델 출력이 `content` 안에 raw 마크업(예: `<tool_call>...`)으로 새어 들어가 §6.2 의 실행 루프가 첫 턴에 `if not ai.tool_calls: break` 로 빠져나오며, "LLM 이 도구를 안 쓰기로 결정한 경우" 와 시그널이 구분되지 않는다.

#### 4.3.1 서빙 매트릭스 (운영 가이드)

| 백엔드 / 모델 | `--enable-auto-tool-choice` | `--tool-call-parser` | parallel tool calls | reasoning 노출 채널 |
| :--- | :--- | :--- | :--- | :--- |
| Hosted OpenAI (gpt-4o, o4-mini) | n/a | n/a | ✓ | o-시리즈는 자동(`reasoning_content`) |
| vLLM + gpt-oss-* | 필수 | `openai` | ✓ | `reasoning_content` (별도 필드) |
| vLLM + Qwen3-Coder-* | 필수 | `qwen3_coder` | 모델/버전별 | `<think>...</think>` 본문 인라인 |
| vLLM + Qwen3-Instruct / Qwen2.5-Instruct | 필수 | `hermes` | 부분 | `<think>...</think>` 본문 인라인 |
| vLLM + Llama-3.x-Instruct | 필수 | `llama3_json` | ✓ | (없음) |
| vLLM + Mistral-Instruct | 필수 | `mistral` | 부분 | (없음) |

> **네이밍 주의:** vLLM 의 `openai` 파서는 "OpenAI hosted API 용" 이 아니라 **"OpenAI harmony 응답 포맷을 쓰는 모델 용"** 으로, 실제 적용 대상은 gpt-oss 다.
> 본 Phase 에서 권장하는 운영 조합: `CHAT_LLM` = Qwen3-Coder (vLLM, `qwen3_coder`), `REASONING_LLM` = gpt-oss-120B (vLLM, `openai`). 두 슬롯 모두 reasoning-capable 모델로 채워질 수 있으므로, §10.2 의 토글 라벨은 모델 성격을 정확히 반영하도록 `CHAT_LLM_LABEL` / `REASONING_LLM_LABEL` 환경변수로 운영자가 자유 변경한다(현재 sample.env 의 "일반 대화" / "심층 추론" 은 hosted OpenAI 기본값일 때의 라벨).

#### 4.3.2 vLLM 인스턴스 검증 절차
운영자가 vLLM 인스턴스를 추가·교체할 때 다음 두 단계를 수행한다.

1. **서버 측 점검**:
   ```bash
   ps -ef | grep -E 'vllm|api_server' | grep -- '--tool-call-parser'
   # 또는
   docker logs <vllm-container> 2>&1 | grep -iE 'tool.?call|auto.?tool.?choice'
   ```
   `--enable-auto-tool-choice` 와 `--tool-call-parser <name>` 둘 다 떠야 한다.

2. **클라이언트 측 sanity check** (도구 1개 + 호출 유도 프롬프트):
   ```bash
   curl -s $BASE_URL/chat/completions -H "Authorization: Bearer $KEY" \
     -H "Content-Type: application/json" -d '{...tools=[{name:"add",...}]...}'
   ```
   응답의 `choices[0].message.tool_calls` 배열이 비어있지 않아야 한다.

#### 4.3.3 런타임 누수 감지 (서비스 가드)
§6.2 의 실행 루프에서 **첫 `ai` 응답이 `tool_calls=[]` 인데** `content` 에 다음 마커 패턴 중 하나가 발견되면, 트레이스에 `warnings: ["tool_call_parser_likely_missing"]` 를 첨부하고 LLM_LOG `type=toollab_tool_parser_leak` 한 줄을 적재한다.

- 누수 패턴(정규식): `<tool_call>`, `<\|tool▁call▁begin\|>`, ``` ```json\s*\{\s*"name" ```, `<function_call>`

이 경고는 "모델이 도구를 안 쓴 것" 과 "vLLM 파서 미설정" 을 운영자가 즉시 구분할 수 있게 해 준다(요구사항 §1 의 비교 가설을 측정할 때 가짜 시그널이 들어오지 않도록 하는 통제 변수).

## 5. 도구 정의 모델 (Tool Definition)

### 5.1 등록 입력 스키마
UI 에서 운영자는 다음 4개 항목을 한 화면에서 입력한다.

```json
{
  "name": "create_user",
  "description": "신규 사용자를 생성한다. 이름·이메일이 필수.",
  "parameters": {
    "type": "object",
    "properties": {
      "name":  {"type": "string", "description": "사용자 이름"},
      "email": {"type": "string", "format": "email"}
    },
    "required": ["name", "email"]
  },
  "returns": {
    "type": "object",
    "properties": {
      "user_id": {"type": "string"},
      "created_at": {"type": "string"}
    }
  },
  "code": "def handler(name: str, email: str):\n    uid = f\"u-{abs(hash(email)) % 100000}\"\n    return {\"user_id\": uid, \"created_at\": \"2026-05-07T00:00:00Z\"}"
}
```

| 필드 | 필수 | 설명 |
| :--- | :--- | :--- |
| `name` | ✓ | LLM 에 노출되는 함수명. 정규식 `^[a-zA-Z_][a-zA-Z0-9_]{0,63}$` 검증. 사용자 범위에서 유일. |
| `description` | ✓ | LLM 이 도구를 선택할 때 참고하는 자연어 설명. 1자 이상. |
| `parameters` | ✓ | JSON Schema (Draft 2020-12 호환). OpenAI tool spec 의 `parameters` 와 동일 형식으로 그대로 LLM 에 전달된다. |
| `returns` | ✓ | 반환값 JSON Schema. **LLM 호출 자체에는 사용되지 않으며**, 함수 실행 후 결과 검증과 UI 표기용. 검증 실패 시 tool 결과를 `error` 로 처리한다. |
| `code` | ✓ | Python 본문. 반드시 **`def handler(...) -> dict|list|str|int|float|bool|None`** 시그니처를 가져야 한다. 시그니처 미존재 시 등록 거부. |
| `is_active` | – | 기본 `true`. 비활성화된 도구는 LLM 카탈로그에 노출되지 않으며 호출되지도 않는다. |
| `tags` | – | 자유 텍스트 라벨(콤마 분리). UI 필터(예: `atomic`, `macro`)용. |

### 5.2 동적 컴파일 규약
- 서버는 등록 시 다음을 순차 검증한다.
  1. JSON Schema (`parameters`, `returns`) 가 `jsonschema` 라이브러리로 컴파일 가능해야 한다.
  2. `code` 가 AST 로 파싱 가능해야 한다.
  3. AST 노드 화이트리스트 검사 (§9 보안).
  4. `handler` 심볼이 정의되어 있어야 하고, 인자 이름들이 `parameters.properties` 의 키와 정확히 일치해야 한다(추가/누락 거부).
- 검증 통과 시 `exec` 로 **격리 네임스페이스**에 컴파일하여 메모리 레지스트리에 저장한다.
- 같은 `name` 으로 다시 등록하면 **버전이 1 증가**하며 이전 버전은 `tool_definitions` 에 보존(`is_active=false`) — 트레이스 재현성 확보.

### 5.3 LangChain Tool 변환
- 메모리 레지스트리는 각 활성 도구를 LangChain `StructuredTool` 로 래핑하여 보관한다.
- LLM 호출 직전 활성 도구 목록을 모아 `llm.bind_tools(tools)` 로 바인딩한다. `bind_tools()` 는 hosted OpenAI / vLLM(gpt-oss, Qwen3-Coder, Llama 등) 백엔드 차이를 흡수하므로 도구 정의 코드는 백엔드별로 분기하지 않는다. (백엔드별 vLLM 파서 요건은 §4.3 참조)
- 모델은 `tool_choice="auto"` (LLM 이 자율 결정) 가 기본. 추후 UI 에서 강제 호출이 필요한 도구를 지정할 수 있는 `tool_choice` 셀렉터를 둔다(필수 아님, P07.1 후보).
  - 단, vLLM 의 `qwen3_coder`, `hermes` 등 일부 파서는 `tool_choice="required"` 또는 특정 함수 강제 모드가 안정적이지 않다. 본 Phase 의 기본인 `auto` 만 정식 지원 범위로 한다.

## 6. 실행 (Run) 모델

### 6.1 입력
| 필드 | 필수 | 설명 |
| :--- | :--- | :--- |
| `prompt` | ✓ | 자연어 사용자 입력. |
| `model_type` | ✓ | `chat` \| `reasoning` (기존 `/chat` 의 model_type 과 동일 의미). |
| `system_prompt` | – | 비어있으면 기본 system 프롬프트 사용. UI 에서 편집 가능. |
| `tool_ids` | – | 명시 시 해당 도구만 노출. 미지정 시 호출자의 모든 활성 도구 노출. |
| `max_tool_iterations` | – | 기본 `5`. tool call 루프 상한. 초과 시 `max_iterations_exceeded` 로 종료. |

### 6.2 실행 루프
```
messages = [system, user(prompt)]
for i in range(max_tool_iterations):
    ai = llm.invoke(messages)                  # tool_calls 가 포함될 수 있음
    record(ai)
    if not ai.tool_calls:
        break
    for call in ai.tool_calls:
        result = execute_tool(call.name, call.args)   # 샌드박스
        messages.append(ToolMessage(result, ...))
        record(call, result)
final_response = ai.content
```
- 루프 상한 도달 시 마지막 AI 응답을 `final_response` 로, 별도 플래그 `truncated=true` 와 함께 반환.
- 기본 동작은 **non-streaming** 으로 충분(트레이스 가독성 우선). 스트리밍은 P07.1 에서 검토.

### 6.3 출력 (트레이스)
```json
{
  "run_id": "...",
  "prompt": "...",
  "model_type": "chat",
  "model": "gpt-4o-2024-08-06",
  "served_by": "openai",
  "tool_call_parser": null,
  "started_at": "...", "ended_at": "...", "latency_ms": 4321,
  "input_tokens": 812, "output_tokens": 124,
  "iterations": 3,
  "truncated": false,
  "warnings": [],
  "steps": [
    {"step": 1, "kind": "ai",   "content": null,
     "reasoning": "(optional) 모델이 reasoning_content 채널로 노출한 경우만 채워짐",
     "tool_calls": [{"id": "c1", "name": "list_users", "args": {}}]},
    {"step": 2, "kind": "tool", "tool_call_id": "c1",
```

루트 필드 보강:
- `served_by`: `openai` (hosted) 또는 `vllm`. `*_BASE_URL` 이 비어있으면 `openai`, 아니면 `vllm`.
- `tool_call_parser`: `served_by=vllm` 일 때만 채워지며 운영자가 env 로 선언한 파서명 라벨(예: `qwen3_coder`, `openai`). hosted OpenAI 면 `null`.
- `warnings`: §4.3.3 의 누수 감지 등 운영 경고 목록(빈 배열 가능). UI 트레이스 푸터에 배지로 노출.

steps 의 ai step 보강:
- `reasoning`: optional. §6.4 참조.

### 6.4 추론 콘텐츠(`reasoning_content`) 캡처
일부 모델은 OpenAI-compatible 응답의 `message.reasoning_content` (LangChain 에서는 `AIMessage.additional_kwargs["reasoning_content"]`) 로 추론 과정을 본문과 분리해 노출한다. 본 Phase 는 해당 채널이 존재하면 트레이스의 ai step 에 `reasoning` 필드로 보존한다.

| 모델 | reasoning 노출 방식 | 본 Phase 처리 |
| :--- | :--- | :--- |
| OpenAI o-시리즈 (o4-mini 등) | `reasoning_content` 별도 필드 | `steps[*].reasoning` 으로 캡처 |
| vLLM + gpt-oss-* (`openai` 파서) | harmony `analysis` 채널 → `reasoning_content` 매핑 | `steps[*].reasoning` 으로 캡처 |
| vLLM + Qwen3 (`<think>...</think>` 인라인) | `content` 본문 안에 thinking 블록 포함 | 본 Phase 에서는 분리 없이 `content` 그대로 보존 (P07.1 에서 분리 검토) |
| 기타 chat 모델 | 노출 없음 | 필드 자체 누락 |

UI(§10.2 우측 패널)는 `reasoning` 이 있는 ai 카드에 "🧠 추론" 접힘 영역(기본 접힘)을 추가한다. 이는 §1 의 비교 가설("reasoning 강한 LLM 은 원자 도구로도 워크플로 조립") 을 정성적으로 보여주는 핵심 디버깅 수단이다.

## 9. 보안 — 동적 코드 실행 가드 (필수)
사용자가 자유 입력하는 Python 코드를 컨테이너 안에서 실행하므로, 본 Phase 에서 가장 위험한 영역이다. 다음 가드를 **모두** 적용한다.

### 9.1 정적 가드 (등록 시점)
- **AST 화이트리스트**: 다음 노드만 허용한다. 그 외(`Import`, `ImportFrom`, `Global`, `Nonlocal`, `Try`(허용), `With`, `ClassDef`(허용), `Yield`/`YieldFrom`(거부), `Lambda`(허용), `AsyncFunctionDef`(거부), `Await`(거부)) — 거부 시 등록 실패.
  - 허용: `FunctionDef`, `Return`, 산술/논리/비교/`If`/`For`/`While`/`Break`/`Continue`/`Pass`/`Assign`/`AugAssign`/`AnnAssign`/`Tuple`/`List`/`Dict`/`Set`/`Subscript`/`Attribute`(아래 sub-rule)/`Call`(아래 sub-rule)/`Name`/`Constant`/`BoolOp`/`UnaryOp`/`Compare`/`IfExp`/`ListComp`/`DictComp`/`SetComp`/`GeneratorExp`/`Try`/`Raise`/`ExceptHandler`/`ClassDef`/`Lambda`.
  - **`Import`/`ImportFrom` 전면 거부.** 모듈 사용은 §9.2 의 사전 주입 네임스페이스로 한정.
  - **`Attribute` 차단 prefix**: `__class__`, `__bases__`, `__subclasses__`, `__globals__`, `__dict__`, `__import__`, `__builtins__`, `__getattribute__`, `func_globals` 등 던더 우회 차단.
  - **`Call` 금지 함수명**: `eval`, `exec`, `compile`, `open`, `input`, `__import__`, `getattr`, `setattr`, `delattr`, `globals`, `locals`, `vars`.
- **JSON Schema 검증**: `parameters`, `returns` 가 `jsonschema` 로 컴파일 가능해야 한다.
- **시그니처 일치**: `handler` 의 인자명/필수 여부가 `parameters` 와 정확히 일치해야 한다.

### 9.2 런타임 가드 (실행 시점)
- **격리 네임스페이스**: `exec(code, sandbox_globals, sandbox_locals)` 의 `sandbox_globals` 에는 `__builtins__` 를 직접 두지 않고, **수동 화이트리스트** (`abs`, `min`, `max`, `sum`, `len`, `range`, `enumerate`, `zip`, `map`, `filter`, `sorted`, `int`, `float`, `str`, `bool`, `list`, `dict`, `set`, `tuple`, `print`(no-op 으로 대체), `round`, `any`, `all`) 만 노출한다.
- **사전 주입 모듈**: `math`, `statistics`, `json`, `re`, `datetime`(`date`, `datetime`, `timedelta`만), `random` 만 모듈 객체로 주입. 그 외 모듈은 코드에서 참조해도 NameError.
- **타임아웃**: 한 번의 `handler` 호출은 기본 **2 초** (env: `TOOLLAB_HANDLER_TIMEOUT_MS`) 이내. 별도 워커 스레드 + `concurrent.futures.ThreadPoolExecutor` 의 `result(timeout=...)` 로 강제 종료(파이썬 스레드 강제 종료 한계 인지 — 응답은 즉시 `timeout` 으로 반환하고, 백그라운드는 완료 시 폐기).
- **결과 검증**: 반환값을 `returns` 스키마로 `jsonschema.validate`. 실패 시 도구 결과를 `{"error": "return_schema_invalid", ...}` 로 LLM 에 전달.
- **로그**: 도구 실행 1 건 = LLM_LOG 1 라인(`type=tool_call_exec`). `name`, `args_redacted=false`, `latency_ms`, `ok`, `error_kind` 포함.

### 9.3 한계 명시 (DoD 외 문서화 항목)
- 본 가드는 **악성 우회 100% 차단을 보장하지 않는다**. 운영 정책상 **Tool Lab 메뉴는 `Admin` 그룹에게만 노출** 하는 것을 기본값으로 한다(`TOOLLAB_ALLOWED_GROUPS=Admin`, env 로 변경 가능). 일반 `User` 가 코드 등록을 못 하도록 막는 1차 방어선이다.
- 외부 네트워크/파일 시스템 접근이 필요한 도구는 **본 Phase 에서 지원하지 않는다.** 향후 별도 컨테이너에 분리한 sidecar runner 로 격리하는 안을 P07.1 에서 검토.

## 10. UI 요구사항

### 10.1 진입점
- 사이드바 `index.html` 에 새 항목 `🛠 Tool Lab` 추가, `/toollab` 으로 라우팅.
- `TOOLLAB_ALLOWED_GROUPS` 에 포함되지 않은 사용자에게는 메뉴를 숨긴다(서버에서 `/api/config` 에 플래그 노출).

### 10.2 페이지 레이아웃 (3 분할)
1. **좌측: 도구 목록 패널**
    - 내 도구의 카드 리스트(이름, description 첫 줄, 태그, 활성 토글, 편집/삭제 아이콘).
    - 상단에 `+ 새 도구` 버튼, 필터 박스(태그/검색).
    - 항목 선택 시 우측 에디터에 로드.
2. **중앙: 도구 에디터**
    - 4 개의 입력 영역: `name`(text), `description`(textarea), `parameters` JSON 에디터, `returns` JSON 에디터, `code` Python 에디터.
    - 코드/스키마 에디터는 **Monaco Editor** (CDN 의존성을 피하기 위해 정적 파일을 `apps/api/static/vendor/monaco/` 에 사전 동봉 — Phase 04 의 Swagger UI 정적 동봉 패턴과 동일). 도입이 무거우면 **CodeMirror 6** 의 정적 번들 또는 단순 `<textarea>` + 신택스 미적용 으로 격하 가능(설계서에서 결정).
    - 하단 액션: `검증`(POST /validate), `저장`(POST/PUT), `취소`.
    - 검증/저장 결과 인라인 표시(에러 라인/메시지).
3. **우측: 실행 패널**
    - 상단: 모델 토글(라벨은 `CHAT_LLM_LABEL` / `REASONING_LLM_LABEL` 환경변수 그대로 노출 — vLLM 운영 환경에서 두 슬롯 모두 reasoning-capable 인 경우 모델명 기반 라벨 권장, 예: `Qwen3-Coder` ↔ `gpt-oss-120B`), 시스템 프롬프트(접힘 가능, 기본은 `tool 사용을 권장하는 짧은 system 메시지`), 도구 멀티셀렉트(기본=현재 활성 모든 도구).
    - 중앙: 자연어 입력창 + `실행` 버튼.
    - 하단: 트레이스 뷰. AI/Tool 단계가 시간순으로 카드로 쌓이며, 각 카드는 펼치면 인자/반환 JSON 을 코드블록으로 표시. ai 카드 중 `reasoning` 필드가 있는 경우(§6.4) "🧠 추론" 접힘 영역(기본 접힘)을 추가로 노출한다.
    - 푸터: `latency_ms`, `iterations`, 토큰 사용량, `truncated` 배지, `model_id` / `served_by` / `tool_call_parser` 라벨, `warnings` 배지(예: `tool_call_parser_likely_missing`).

### 10.3 시드 도구 (첫 기동 시 자동 등록)
운영자가 즉시 데모할 수 있도록 다음을 시드한다(`tags=["seed","atomic"]`, owner = `system`).

| name | 설명 | 비고 |
| :--- | :--- | :--- |
| `list_users` | 메모리 상의 가짜 사용자 목록 반환 | atomic |
| `get_user` | id 로 사용자 1명 조회 | atomic |
| `create_user` | 사용자 생성(메모리) | atomic |
| `update_user_email` | 이메일 갱신 | atomic |
| **`onboard_new_team`** | 부서/이름 리스트를 받아 사용자 생성 + 그룹 할당 + 환영 메시지 작성을 한 번에 수행 | **macro** — 추론 약한 LLM 도 한 방에 끝낼 수 있음 |

> 시드 도구의 백엔드 상태는 **프로세스 인-메모리 dict** 만 사용한다(컨테이너 재기동 시 초기화 — 학습용으로 충분, DB 마이그레이션 회피).

**공통 라벨 (모든 `toollab_*` 라인에 추가)**
- `model_id`: 실제 모델 ID (`/v1/models` 첫 결과). 예: `gpt-4o-2024-08-06`, `Qwen3-Coder-30B-A3B`, `gpt-oss-120b`. Phase 06 의 기존 `model` 필드와는 별개로, 모델 동등성 비교를 위한 정규화 라벨이다.
- `served_by`: `openai` (hosted) / `vllm`. `*_BASE_URL` 이 비어있으면 `openai`.
- `tool_call_parser`: `served_by=vllm` 일 때만. `CHAT_LLM_TOOL_PARSER` / `REASONING_LLM_TOOL_PARSER` env 값을 그대로 라벨링.

- Promtail 의 기존 `[LLM_LOG]` 추출 파이프라인을 **변경 없이 그대로 사용**(접두사·JSON 구조 동일).
- Grafana 에 사전 프로비저닝 대시보드 1종 **"Tool Lab"** 추가:
  - 시간대별 run 수, tool call 수
  - 도구별 호출 빈도 Top 20
  - 평균 iterations / truncated 비율
  - **`model_id` × `tool_call_parser` 별 평균 iterations·latency 비교** (응용력 비교 핵심 패널 — 모델 능력과 파서 설정의 영향을 분리해 볼 수 있게 함)
  - 도구 실행 에러 현황 (`ok=false` count + error_kind 분포)
  - **`toollab_tool_parser_leak` 카운트** (운영자 알림용 — 0 이 아니면 vLLM 인스턴스 설정 점검 필요)

## 13. 호환성 / 비기능 요구사항
- **백엔드 API 시그니처 변경 금지**: `/chat`, `/api/rag/*`, `/api/prompts/*` 의 입출력 형식은 변경하지 않는다. 신규 라우터는 모두 `/api/toollab` 하위.
- **회귀 영향 없음**: `TOOLLAB_ENABLED=false` 로 두면 메뉴/엔드포인트 모두 비노출. Phase 01~06 의 기존 회귀 테스트는 그대로 통과해야 한다.
- **확장성**: 도구 정의는 `(owner_user_id, name, version)` 으로 영속되어, 추후 멀티테넌트/공유 카탈로그로 확장 시 데이터 마이그레이션 없이 가능.
- **성능**: 단일 run 의 LLM 호출 외 오버헤드는 100 ms 이하(샌드박스 컴파일은 등록 시 1회만, 실행은 이미 컴파일된 `code` 객체 재사용).

## 14. 테스트 전략
- **단위 테스트** (pytest 신규 ~28건):
  - AST 검증기: 금지 노드/우회 패턴 케이스 표(예: `__class__.__bases__` 차단, `getattr` 차단, 무한루프 타임아웃, `import os` 거부).
  - 시그니처 매칭: 누락/추가 인자 거부.
  - JSON Schema 검증: 잘못된 스키마 거부.
  - 결과 스키마 위반 시 LLM 에 `error` 페이로드 전달.
  - **§4.3.3 파서 누수 감지**: gpt-oss 누수 샘플(`<tool_call>...`), Qwen3 누수 샘플(`<|tool▁call▁begin|>...`), 일반 텍스트(false positive 0) 3종 케이스 표.
  - **§6.4 reasoning 캡처**: `AIMessage.additional_kwargs["reasoning_content"]` 가 있을 때 트레이스 step 에 `reasoning` 필드로 흐르는지 / 없을 때 필드 자체가 누락되는지.
  - **라벨링**: `*_BASE_URL` 비어있을 때 `served_by=openai` / 채워졌을 때 `served_by=vllm` 로 트레이스·LLM_LOG 양쪽에 찍히는지.
- **통합 테스트** (~5건):
  - 도구 등록 → 활성 → 자연어 1회 입력 → 트레이스에 tool_calls 포함 확인 (LLM 은 OpenAI 함수콜 모킹).
  - 같은 자연어 입력 + (a) 원자만 vs (b) 매크로 포함 의 트레이스 차이 회귀(매크로 활성 시 iterations 가 더 적게 나오는지).
  - 모킹 LLM 이 `tool_calls=[]` + `content="<tool_call>{...}</tool_call>"` 응답을 1회 줬을 때 `warnings` 에 `tool_call_parser_likely_missing` 가 들어가고 LLM_LOG `toollab_tool_parser_leak` 라인 1건 발생 확인.
- **회귀**: Phase 01~06 의 기존 pytest 전체 통과 (현재 77건).
