# Phase 3 상세 설계서 (Data & RAG - 기반 인프라)

## 1. 시스템 아키텍처

### 1.1 계층 구조 (SOLID 원칙 적용)
```
[FastAPI Router]  ← HTTP 요청/응답만 처리 (SRP)
       ↓
[RAGService]      ← 비즈니스 로직 + Qdrant 연동 (SRP, DIP)
       ↓
[Repository]      ← DB CRUD 전용, Generic Base (SRP, OCP, LSP)
       ↓
[SQLAlchemy]      ← ORM / AsyncSession
       ↓
[PostgreSQL]      ← 영속 데이터 저장소
```

### 1.2 프론트엔드 구조 (SRP 적용)
```
static/
├── css/style.css              ← 전체 스타일시트
├── js/
│   ├── app.js                 ← 초기화, 인증, 설정 로드
│   ├── chat.js                ← 채팅 송수신, 스트리밍
│   └── admin.js               ← 모달 동적 로드 + CRUD
├── partials/
│   └── admin-modal.html       ← 관리자 팝업 마크업
├── index.html                 ← 채팅 메인 화면
└── unauthorized.html          ← 비인가 안내 페이지
```

## 4. 핵심 모듈 설계

### 4.1 BaseRepository (`libs/core/repository.py`)
- Generic[T] 기반의 범용 CRUD Repository.
- `get_all()`, `get_by_id()`, `create()`, `update()`, `delete()` 제공.
- `CollectionRepository`, `DomainRepository`가 상속하여 사용.

### 4.2 RAGService (`libs/core/service.py`)
- `CollectionRepository` + `DomainRepository`를 합성(Composition)하여 사용.
- 콜렉션 삭제 시 `QdrantClient`를 통해 Vector DB 콜렉션도 함께 삭제 처리.
- FastAPI `Depends(get_rag_service)`로 주입.

### 4.3 Database (`libs/core/database.py`)
- `create_async_engine` + `async_sessionmaker` 구성.
- `get_db()` 제너레이터로 세션 라이프사이클 관리.
