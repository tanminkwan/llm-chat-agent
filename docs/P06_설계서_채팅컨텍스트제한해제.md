# Phase 6 설계서 - 채팅 API 설계 결함 수정 (GET -> POST 전환)

## 1. 개요 및 변경 대조 (AS-IS vs TO-BE)

기존 시스템의 잘못된 데이터 전송 설계(URL Query Parameter 방식)로 인해 발생하는 `414 Request-URI Too Large` 에러 및 대규모 컨텍스트 전송 불능 결함을 수정한다.

### 1.1 비교표
| 구분 | AS-IS (기존) | TO-BE (변경 후) |
| :--- | :--- | :--- |
| **HTTP Method** | `GET` (또는 Query Param 포함 POST) | **`POST`** |
| **데이터 위치** | **URL Query Parameter** | **JSON Request Body** |
| **전송 한계** | **약 8KB** (Nginx/브라우저 제한) | **제한 없음** (MB 단위 처리 가능) |
| **주요 에러** | 414 Request-URI Too Large | 없음 |

### 1.2 호출 예시 비교
- **AS-IS**: URL에 데이터가 모두 노출되며 길이가 길어지면 서버가 거부함
  ```http
  GET /chat?message=안녕&system_prompt=너는... (데이터가 길어지면 에러 발생)
  ```
- **TO-BE**: 데이터가 HTTP 본문에 숨겨져 안전하고 대용량 전송 가능
  ```http
  POST /chat
  Content-Type: application/json

  {
    "message": "안녕 (매우 긴 텍스트 입력 가능)",
    "system_prompt": "너는...",
    "temperature": 0.7
  }
  ```

## 2. 변경 설계

### 2.1 데이터 전송 규격 (API 설계)
- **엔드포인트**: `POST /chat`
- **Content-Type**: `application/json`
- **Request Body (JSON)**:
    - `message` (string, 필수): 사용자 메시지
    - `model_type` (string, 선택): "chat" 또는 "reasoning"
    - `system_prompt` (string, 선택): 시스템 지침
    - `temperature` (float, 선택): 창의성 (0~1)
    - `session_id` (string, 선택): 대화 세션 식별자

### 2.2 프론트엔드 연동
- `apps/api/static/js/chat.js`의 `sendMessage` 함수를 수정하여 `fetch` 요청 시 `method: 'POST'` 및 JSON `body`를 사용하도록 변경한다.

### 2.3 백엔드 구현
- `apps/api/schemas.py`에 `ChatRequest` Pydantic 모델을 정의한다.
- `apps/api/main.py`의 `chat` 함수를 `ChatRequest` 바디를 받도록 수정한다.

## 4. 로깅 설계 (Structured JSON Logging)

ELK/Prometheus 등 로그 수집 시스템과의 연동을 위해 모든 LLM 통신 내역을 구조화된 JSON 형식으로 기록한다.

### 4.1 로깅 원칙
- **접두사**: 로그 필터링을 위해 모든 관련 로그에 `[LLM_LOG]` 접두사를 부여한다.
- **포맷**: 한 줄의 유효한 JSON 객체로 출력한다 (`json.dumps` 사용).
- **분리**: 추적성을 위해 요청(Request)과 응답(Response) 로그를 별도로 기록하며, `request_id`로 상호 연결한다.

### 4.2 데이터 규격
- **공통 필드**: `request_id`, `user_id`, `type` (request/response/rag_search_request/rag_search_response)
- **RAG 검색 로그**: 대용량 텍스트(`content`)는 제외하고 메타데이터(`id`, `score`, `source` 등)만 포함하여 추적성을 보장한다.
