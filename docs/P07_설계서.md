## 0. 검증 결과 반영 요약 (§VAL → 본 설계 입력)

| :--- | :--- | :--- |

## 2. 데이터 모델

### 2.1 DDL — `tool_definitions`

```sql
CREATE TABLE tool_definitions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_user_id   TEXT NOT NULL,
  name            TEXT NOT NULL CHECK (name ~ '^[a-zA-Z_][a-zA-Z0-9_]{0,63}$'),
  version         INTEGER NOT NULL DEFAULT 1 CHECK (version >= 1),
  description     TEXT NOT NULL,
  parameters_json JSONB NOT NULL,
  returns_json    JSONB NOT NULL,
  code            TEXT NOT NULL,
  tags            TEXT[] NOT NULL DEFAULT '{}',
  is_active       BOOLEAN NOT NULL DEFAULT TRUE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (owner_user_id, name, version)
);

CREATE INDEX ix_tool_definitions_owner_active
  ON tool_definitions (owner_user_id, is_active, name);
CREATE INDEX ix_tool_definitions_tags
  ON tool_definitions USING gin (tags);
```

> **버전 관리**: 같은 `(owner_user_id, name)` 으로 PUT 시
> ① 기존 row 의 `is_active=false` 로 deactivate
> ② `version+1` 인 새 row 를 `is_active=true` 로 INSERT
> 두 작업을 단일 트랜잭션. UI 는 항상 활성 row 만 노출하지만,
> `tool_runs.tool_ids` 는 PK 로 참조하므로 트레이스 재현성 유지.

## 3. 도구 정의 등록 흐름

### 3.1 등록 API 핸들러 흐름 (POST/PUT)

```
[HTTP 요청] (§RQ 5.1 의 5필드 + tags)
   │
   ▼
[Pydantic 검증] schemas.ToolDefinitionInput
   │  ├── name 정규식
   │  └── parameters/returns 가 dict 인지 (JSON Schema 자체 검증은 다음 단계)
   ▼
[JSON Schema 컴파일] jsonschema.validators.Draft202012Validator.check_schema(...)
   │  └── 실패 → 400 + errors[]
   ▼
[AST 화이트리스트 검증] sandbox.validate_code_ast(code)
   │  └── 실패 → 400 + 노드 위반 위치
   ▼
[handler 시그니처 매칭] sandbox.compile_and_check_signature(code, parameters)
   │  ├── exec 임시 네임스페이스에서 컴파일
   │  ├── handler 심볼 존재 + 인자명/필수여부 일치
   │  └── 실패 → 400
   ▼
[DB 업서트] (트랜잭션)
   │  ├── 같은 (owner, name) 활성 row 가 있으면 deactivate + version+1
   │  └── 새 row INSERT
   ▼
[Registry 갱신] registry.replace_active(tool_def)
   │  └── StructuredTool 래핑 + handler closure 캐시
   ▼
[LLM_LOG] toollab_tool_register
   │
   ▼
[201 Created or 200 OK]
```

### 3.2 검증 단계의 trade-off

- **POST `/validate`** 는 위 흐름 중 **DB 업서트 전까지** 만 수행 (`registry` 도 미갱신).
  UI 의 `검증` 버튼이 누름.
- 모든 검증 실패는 **첫 실패에서 멈춰서** 사유 1건 + 위치 정보를 돌려준다.
  실시간 UI 의 메시지 명확성을 위해 multi-error 누적은 하지 않음.

### 3.3 동시성

- 같은 사용자가 같은 name 을 동시에 등록하는 race 는 `(owner_user_id, name, version)`
  UNIQUE 제약 + `SERIALIZABLE` 트랜잭션으로 한쪽 실패. 실패 시 클라이언트가 재시도.
- Registry in-memory 갱신은 `threading.Lock` 으로 단일 프로세스 직렬화. 워커 다중화
  (uvicorn `--workers > 1`) 환경은 본 Phase 외 — 단일 워커 가정.

## 4. 동적 컴파일 + 보안 가드

### 4.1 AST 화이트리스트 검증기

```python
# libs/toollab/sandbox.py (요약)

import ast
from dataclasses import dataclass
from typing import Iterable

ALLOWED_NODES: frozenset[type[ast.AST]] = frozenset({
    ast.Module, ast.FunctionDef, ast.arguments, ast.arg, ast.Return,
    ast.Assign, ast.AugAssign, ast.AnnAssign, ast.Expr,
    ast.If, ast.For, ast.While, ast.Break, ast.Continue, ast.Pass,
    ast.Tuple, ast.List, ast.Dict, ast.Set,
    ast.Subscript, ast.Index, ast.Slice,
    ast.Attribute, ast.Call, ast.Name, ast.Constant,
    ast.BoolOp, ast.UnaryOp, ast.BinOp, ast.Compare, ast.IfExp,
    ast.ListComp, ast.DictComp, ast.SetComp, ast.GeneratorExp, ast.comprehension,
    ast.Try, ast.Raise, ast.ExceptHandler, ast.ClassDef, ast.Lambda,
    ast.And, ast.Or, ast.Not, ast.USub, ast.UAdd,
    ast.Add, ast.Sub, ast.Mult, ast.Div, ast.Mod, ast.Pow, ast.FloorDiv,
    ast.Eq, ast.NotEq, ast.Lt, ast.LtE, ast.Gt, ast.GtE, ast.In, ast.NotIn, ast.Is, ast.IsNot,
    ast.Load, ast.Store, ast.Del, ast.Param,
    ast.keyword, ast.alias, ast.Starred,
})

# 명시 거부 (allowlist 의 보충 — 가독성 위해 분리)
DENIED_NODES: frozenset[type[ast.AST]] = frozenset({
    ast.Import, ast.ImportFrom, ast.Global, ast.Nonlocal, ast.With,
    ast.AsyncFunctionDef, ast.Await, ast.AsyncFor, ast.AsyncWith,
    ast.Yield, ast.YieldFrom,
})

DENIED_ATTR_PREFIXES = (
    "__class__", "__bases__", "__subclasses__", "__globals__", "__dict__",
    "__import__", "__builtins__", "__getattribute__", "__getattr__",
    "__setattr__", "__delattr__", "__mro__", "__init_subclass__",
    "func_globals", "f_globals", "f_locals",
)

DENIED_CALL_NAMES = frozenset({
    "eval", "exec", "compile", "open", "input", "__import__",
    "getattr", "setattr", "delattr", "globals", "locals", "vars",
    "memoryview", "breakpoint", "help",
})

@dataclass
class AstViolation:
    line: int
    col: int
    kind: str
    detail: str

def validate_code_ast(code: str) -> list[AstViolation]:
    tree = ast.parse(code)  # SyntaxError 는 외부에서 처리
    violations: list[AstViolation] = []
    for node in ast.walk(tree):
        cls = type(node)
        if cls in DENIED_NODES:
            violations.append(AstViolation(
                node.lineno, node.col_offset,
                "denied_node", cls.__name__))
        elif cls not in ALLOWED_NODES:
            violations.append(AstViolation(
                node.lineno, node.col_offset,
                "unknown_node", cls.__name__))
        if isinstance(node, ast.Attribute):
            if any(node.attr.startswith(p) for p in DENIED_ATTR_PREFIXES):
                violations.append(AstViolation(
                    node.lineno, node.col_offset,
                    "denied_attr", node.attr))
        if isinstance(node, ast.Call):
            fn = node.func
            if isinstance(fn, ast.Name) and fn.id in DENIED_CALL_NAMES:
                violations.append(AstViolation(
                    node.lineno, node.col_offset,
                    "denied_call", fn.id))
    return violations
```

### 4.3 컴파일 + 시그니처 매칭

```python
def compile_and_check_signature(code: str, parameters: dict) -> "HandlerCallable":
    sb_globals = build_sandbox_globals()
    sb_locals: dict = {}
    compiled = compile(code, filename="<toollab>", mode="exec")
    exec(compiled, sb_globals, sb_locals)  # 격리 네임스페이스
    handler = sb_locals.get("handler")
    if not callable(handler):
        raise ToolDefinitionError("handler_not_defined")

    import inspect
    sig = inspect.signature(handler)
    expected_keys = set(parameters.get("properties", {}).keys())
    expected_required = set(parameters.get("required", []))
    actual_keys = set(sig.parameters.keys())
    actual_required = {
        name for name, p in sig.parameters.items()
        if p.default is inspect.Parameter.empty
    }
    if actual_keys != expected_keys:
        raise ToolDefinitionError(f"args_mismatch: {actual_keys ^ expected_keys}")
    if actual_required != expected_required:
        raise ToolDefinitionError(
            f"required_mismatch: handler={actual_required} schema={expected_required}")
    return handler
```

### 4.4 타임아웃 적용

```python
# libs/toollab/sandbox.py (이어서)

from concurrent.futures import ThreadPoolExecutor, TimeoutError as FutTimeoutError

_EXECUTOR = ThreadPoolExecutor(max_workers=4, thread_name_prefix="toollab-handler")
# max_workers 4: 동시 실행 도구 수 상한 (요청 당 평균 1~2 도구 가정)

def run_handler_with_timeout(handler, kwargs: dict, timeout_ms: int) -> dict:
    fut = _EXECUTOR.submit(handler, **kwargs)
    try:
        return fut.result(timeout=timeout_ms / 1000)
    except FutTimeoutError:
        # Python 스레드 강제 종료 한계 인지 — 응답은 즉시 반환,

        raise ToolExecutionError("timeout", timeout_ms=timeout_ms)
```

## 5. 실행 루프 (Tool Calling Engine)

### 5.1 LangChain bind_tools 통합

- `libs/core/llm.py` 의 기존 `get_chat_llm(model_type)` 그대로 사용. 백엔드별 분기 없음.
- `ToolRunner` 가 `registry.list_active_tools_for(user, tool_ids)` 로 `StructuredTool` 목록을
  받아 `llm.bind_tools(tools, tool_choice="auto", parallel_tool_calls=True)` 로 LLM 만든다.
- StructuredTool 의 `args_schema` 는 `parameters` (JSON Schema) 를 Pydantic v2 모델로
  컴파일한 결과를 사용 (`pydantic.create_model` + `jsonschema_to_pydantic` 자체 헬퍼).

### 5.2 실행 루프 의사 코드

```python
# libs/toollab/runner.py (요약)

from langchain_core.messages import SystemMessage, HumanMessage, ToolMessage
from .serializer import to_tool_message_content
from .leak_detector import detect_leak

def run(req: RunRequest) -> RunResult:
    tools = registry.list_active_tools_for(req.user, req.tool_ids)
    llm = get_chat_llm(req.model_type).bind_tools(
        tools, tool_choice="auto", parallel_tool_calls=True
    )

    messages = []
    if req.system_prompt:
        messages.append(SystemMessage(req.system_prompt))
    else:
        messages.append(SystemMessage(DEFAULT_SYSTEM_PROMPT))
    messages.append(HumanMessage(req.prompt))

    trace = TraceAssembler(req)
    iterations = 0
    truncated = False
    max_iter = min(req.max_tool_iterations or DEFAULT_MAX_ITER,
                   TOOLLAB_MAX_TOOL_ITERATIONS_HARD)

    while iterations < max_iter:
        iterations += 1
        ai = llm.invoke(messages)              # AIMessage
        trace.record_ai(ai)
        messages.append(ai)

        # §6.2 누수 감지 — tool_calls 비었는데 raw 마커가 보이면 경고
        if not ai.tool_calls:
            leak = detect_leak(ai.content or "")
            if leak:
                trace.add_warning("tool_call_parser_likely_missing", leak)
            break

        for tc in ai.tool_calls:
            exec_result = registry.execute(
                tool_name=tc["name"], args=tc["args"], user=req.user
            )  # returns: {"ok": bool, "result"|"error": ..., "latency_ms": int}
            content = to_tool_message_content(exec_result)   # §5.3
            messages.append(ToolMessage(
                content=content,
                tool_call_id=tc["id"],
                name=tc["name"],
            ))
            trace.record_tool(tc, exec_result)
    else:
        truncated = True   # while 의 else — 루프 exit 없이 max 도달 시

    return trace.finalize(iterations=iterations, truncated=truncated)
```

### 5.3 ToolMessage 직렬화 정책 — **§VAL T5c 발견 반영 (가장 중요한 신규 결정)**

§VAL T5c 에서 확인된 사실: **vLLM 의 일부 chat template / tool-call-parser 가
escape 된 JSON 배열을 tool 메시지 content 에 박으면 400 으로 거절**한다.
즉 handler 의 결과를 그대로 `json.dumps(result)` 해서 ToolMessage.content 에 넣으면
운영 중 silent 400 의 위험이 있다.

본 설계는 다음 정책을 강제한다:

```python
# libs/toollab/serializer.py

import json
from typing import Any

def to_tool_message_content(exec_result: dict) -> str:
    """
    handler 실행 결과를 ToolMessage.content (string) 로 변환.
    §VAL T5c 정책: 배열·객체 등 구조화 데이터는 'object wrapping' 으로 한 단계 감싸,
                   chat template 이 raw JSON 배열로 인식하지 않도록 한다.
    """
    if not exec_result.get("ok", False):
        # 에러는 항상 객체 wrapping
        return json.dumps({
            "ok": False,
            "error": exec_result.get("error"),
            "error_kind": exec_result.get("error_kind"),
        }, ensure_ascii=False)

    result = exec_result.get("result")
    # 정책 1 — 단순 스칼라는 그대로 string 화 (자연어와 충돌 없음)
    if isinstance(result, (str, int, float, bool)) or result is None:
        return json.dumps({"ok": True, "result": result}, ensure_ascii=False)

    # 정책 2 — list / dict 는 항상 {"ok": True, "result": ...} 객체 wrapping
    # 핵심: tool content 의 최상위가 '[' 으로 시작하지 않도록 보장
    return json.dumps({"ok": True, "result": result}, ensure_ascii=False)
```

> **불변식 (T5c 발견의 직접 대응)**: `to_tool_message_content` 의 반환값은 **항상 `{`
> 으로 시작** 한다 — 즉 최상위가 JSON object. JSON 배열을 string 화한 결과를 직접 박는
> 경로는 코드상 존재하지 않는다.
>
> 운영 중 chat template 이 객체 wrapping 마저 거절하는 새 케이스가 발견되면, 본 모듈만
> 수정해 자연어 1줄 fallback (`f"Tool returned: {len(result)} items"` 류) 를 추가하면
> 호출부 변경 없이 정책 갱신 가능.

### 5.4 max_iterations 가드

- 기본값: `TOOLLAB_MAX_TOOL_ITERATIONS=8` (§0 의 §VAL 반영 — sequential 모델의 worst case
  여유 확보. §RQ 의 5 보다 상향). request 별 override 허용.
- hard cap: 환경변수 `TOOLLAB_MAX_TOOL_ITERATIONS_HARD=16` — request override 가
  이 값을 넘지 못한다 (악성 prompt 방어).
- 도달 시 `truncated=true` + 마지막 AI 응답을 final_response 로. 추가 호출은 안 함.

### 6.2 tool_call_parser 누수 감지

```python
# libs/toollab/leak_detector.py

import re
from typing import NamedTuple

# §RQ 4.3.3 의 정규식 — §VAL T9 결과 도착 시 보강
_LEAK_PATTERNS: list[tuple[str, re.Pattern]] = [
    ("xml_tool_call",    re.compile(r"<tool_call\b")),
    ("qwen_tool_call",   re.compile(r"<\|tool▁call▁begin\|>")),
    ("json_name_block",  re.compile(r"```json\s*\{\s*\"name\"")),
    ("function_call",    re.compile(r"<function_call\b")),
    ("harmony_channel",  re.compile(r"<\|channel\|>(commentary|analysis)")),
]

class LeakHit(NamedTuple):
    pattern: str
    excerpt: str

def detect_leak(content: str) -> LeakHit | None:
    if not content:
        return None
    for name, rx in _LEAK_PATTERNS:
        m = rx.search(content)
        if m:
            start = max(0, m.start() - 20)
            end = min(len(content), m.end() + 200)
            return LeakHit(pattern=name, excerpt=content[start:end])
    return None
```

누수 감지 시:
- 트레이스 `warnings.append("tool_call_parser_likely_missing")`
- LLM_LOG `type=toollab_tool_parser_leak` (`pattern_matched`, `content_excerpt` 포함)

### 6.3 트레이스 어셈블리

`TraceAssembler` 는 §RQ 6.3 출력 그대로 produce. 추가:
- `iterations` = 실행 루프의 turn 카운트 (§VAL §6.0 정의 — turn 수)
- 각 step 의 latency 는 step 사이의 시간차로 산출 (LLM call 또는 tool 실행 1건 단위)
- `usage_metadata` 가 final AI message 에 있으면 `input_tokens`/`output_tokens` 갱신.
  reasoning_tokens 가 별도 노출되는 vLLM 빌드는 `output_tokens` 합계에 포함 (§VAL T10
  결과 후 정책 재확정).

## 부록 A — 본 설계의 §VAL 의존성 핫스팟 요약

본 설계의 다음 결정은 §VAL 의 **확정 결과** 에 직접 의존한다. 결과가 갱신되면 본 설계의
해당 섹션을 동시에 갱신해야 한다 (drift 방지).

| 본 설계 항목 | 의존 §VAL 항목 | 현재 상태 |
| :--- | :--- | :--- |
| §5.3 ToolMessage 직렬화 정책 (object wrapping) | T5c 부수 발견 (escape JSON → 400) | ✅ 확정 — 본 설계의 hard 제약 |
| §5.4 max_iterations 기본값 (5→8) | T4b sequential iterations 실측 | ⏳ 미입력 — 8 은 잠정값 |
| §6.1 reasoning 캡처 (양 채널 처리) | T7 양 모델 결과 | ⏳ T2 시점 부분 확인, T7 정식 결과 대기 |
| §6.2 누수 패턴 5종 | T9 결과 | ⏳ 미입력 — §RQ 4.3.3 그대로 |
| §13 단위 테스트의 T9 모방 케이스 | T9 결과 | ⏳ 미입력 |
| §5.2 inner-loop 보존 | T4 (Qwen parallel ✅) | ✅ 확정 |
| §3 등록 흐름의 returns 검증 | T8 결과 | ⏳ 미입력 — args 검증 추가는 P07.1 |

> 운영 측 검증이 끝나는 대로 본 설계 §0 표 + 본 부록을 갱신해 drift 를 방지한다.
