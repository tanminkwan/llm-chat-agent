# Phase 3 요구사항 정의서 (Data & RAG)

## 1. 개요
본 단계에서는 RAG 시스템의 기반이 되는 **데이터 관리 인프라**를 구축합니다.
PostgreSQL을 도입하여 콜렉션과 도메인 메타데이터를 관리하고, Admin 전용 CRUD UI를 제공합니다.

## 2. 목표
- PostgreSQL 데이터베이스 도입 및 연동
- RAG 콜렉션(Collection) 메타데이터 CRUD 구현
- 지식 도메인(Domain) 분류 체계 CRUD 구현
- Admin 전용 관리 UI 구현

## 3. 상세 요구사항

### 3.1 데이터베이스 (PostgreSQL)
- **인프라**: Docker Compose에 `postgres:16-alpine` 서비스 추가.
- **ORM**: SQLAlchemy 2.0 (Async) + asyncpg 드라이버.
- **테이블 자동 생성**: 앱 시작 시 `Base.metadata.create_all` 실행.
- **설정 관리**: `DATABASE_URL`을 `.env`/`Settings`로 관리. `sample.env` 동기화 필수.

### 3.2 Collection 관리
- **모델 필드**: id, name(unique), description, snippet_size_limit, search_method.
- **검색 방식**: `vector` (Vector Distance), `bm25` (키워드), `hybrid` (복합).
- **CRUD**: 생성/조회/수정/삭제 API (`/api/collections`).
- **삭제 시**: Vector DB(Qdrant) 내 동명 콜렉션 삭제 여부를 사용자에게 확인.

### 3.3 Domain 관리
- **모델 필드**: id, name(unique).
- **용도**: 과학, 경제, IT 등 지식 분야 분류.
- **CRUD**: 생성/조회/수정/삭제 API (`/api/domains`).

### 3.4 Admin UI
- **권한**: Admin 그룹 사용자에게만 사이드바에 관리 메뉴 노출.
- **모달 팝업**: 콜렉션/도메인 각각 등록 폼 + 목록 테이블을 2-column 레이아웃으로 표시.
- **인라인 편집**: 목록에서 행 클릭 시 폼에 데이터가 로드되어 수정 가능.
- **삭제 확인**: 콜렉션 삭제 시 Vector DB 데이터 삭제 여부를 묻는 confirm 팝업 표시.

### 3.5 아키텍처 (SOLID 원칙)
- **Repository 패턴**: `libs/core/repository.py` - DB CRUD 전용 계층.
- **Service 패턴**: `libs/core/service.py` - 비즈니스 로직 (Qdrant 연동 포함).
- **프론트엔드 분리**: HTML(구조) / CSS(스타일) / JS(동작) / Partials(팝업) 파일 분할.
