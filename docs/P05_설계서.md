# Phase 05 설계서 (Custom Embedding API Support)

## 1. 아키텍처 개요
환경 변수 `EMBEDDING_USE_CUSTOM` 플래그를 도입하여, 기존 OpenAI API 방식과 프로덕션 전용 커스텀 API 방식 간의 전환을 명시적으로 제어함. 이를 통해 특정 모델명에 종속되지 않는 범용적인 구조를 확보함.

## 2. 상세 설계

### 2.1 CustomProductionEmbeddings 클래스 설계
프로덕션의 단순화된 API 규격을 처리하는 범용 클래스임.

- **속성**:
  - `base_url`: `settings.EMBEDDING_BASE_URL`
  - `model`: `settings.EMBEDDING_MODEL`
- **메서드**:
  - `embed_documents(texts)`: `httpx`를 사용하여 `{"tests": texts}` 페이로드 전송.
  - `embed_query(text)`: `embed_documents([text])` 호출.
  - 비동기 대응: `aembed_documents`, `aembed_query` 구현.

### 2.2 LLMGateway 분기 로직 수정
`get_embeddings()` 메서드는 모델명이 아닌 **플래그**를 기준으로 판단함.

```python
def get_embeddings() -> Embeddings:
    # 플래그가 true인 경우 커스텀 방식 사용
    if settings.EMBEDDING_USE_CUSTOM:
        return CustomProductionEmbeddings(
            base_url=settings.EMBEDDING_BASE_URL,
            model=settings.EMBEDDING_MODEL
        )
    
    # 기본값 (false) 또는 설정 없을 시 기존 OpenAI 방식 사용
    return OpenAIEmbeddings(...)
```

### 2.3 환경 설정 변경 (libs/core/settings.py)
- `EMBEDDING_USE_CUSTOM: bool = False` 필드 추가.
- `Pydantic Settings`를 통해 `.env`에서 값을 읽어오도록 구성.

## 3. 데이터 흐름
1. 애플리케이션 기동 시 `.env`의 `EMBEDDING_USE_CUSTOM` 값 로드.
2. `LLMGateway.get_embeddings()` 호출 시 해당 플래그 확인.
3. `True`이면 `CustomProductionEmbeddings`, `False`이면 `OpenAIEmbeddings` 객체 생성 및 반환.
4. 이후 과정은 기존 RAG 프로세스와 동일.

## 4. 예외 처리 및 고려 사항
- 플래그가 `True`임에도 `EMBEDDING_BASE_URL`이 설정되지 않은 경우에 대한 방어 로직 고려.
- `httpx` 클라이언트의 커넥션 풀링 및 재사용성 검토 (범용성 향상).

## 5. 테스트 계획
- **Mock 서버 기반 통합 테스트**:
  - `httpx` 요청을 가로채는 Mock 서버를 구축하여 테스트 수행.
  - 테스트 코드에서 `{"tests": ["hello"]}` 요청을 보낼 때, 가상 서버가 `{"embeddings": [[...]]}`를 반환하도록 시뮬레이션.
  - `CustomProductionEmbeddings` 클래스가 해당 응답을 정확히 리스트 형태로 변환하는지 검증.
- **분기 로직 테스트**:
  - `EMBEDDING_USE_CUSTOM` 설정에 따라 `OpenAIEmbeddings`와 `CustomProductionEmbeddings`가 적절히 교체되는지 확인.
- **비동기 테스트**:
  - `aembed_query`, `aembed_documents` 메서드가 비동기 환경에서 가상 서버와 정상 통신하는지 검증.
