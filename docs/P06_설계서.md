## 2. llm-agent 로그 보강 설계

### 2.1 신규 필드 정의

| 필드 | 타입 | 적용 type | 산출 방법 |
| :--- | :--- | :--- | :--- |
| `timestamp` | string (ISO 8601 UTC) | 모든 LLM_LOG | `datetime.now(timezone.utc).isoformat()` 를 페이로드 직전에 기록 |
| `latency_ms` | int | `response`, `error`, `rag_search_response` | `request_started_at` 과의 차 (ms) |
| `model` | string | `response` | langchain `BaseChatModel.model_name` 또는 chunk metadata 의 `model` 필드 |
| `input_tokens` | int \| null | `response` | 마지막 chunk 의 `usage_metadata.input_tokens` (없으면 null) |
| `output_tokens` | int \| null | `response` | 마지막 chunk 의 `usage_metadata.output_tokens` |
| `top_score` | float \| null | `rag_search_response` | `max(score for r in results)`; vector 검색 외에는 null |
| `min_score` | float \| null | `rag_search_response` | `min(score for r in results)`; vector 검색 외에는 null |

### 2.2 구현 패턴

기존 `[LLM_LOG]` 빌드 코드를 직접 수정하지 않고, 작은 헬퍼를 도입해 *모든 LLM_LOG 라인에 자동으로 timestamp 가 포함되도록* 한다.

```python
# apps/api/main.py (또는 libs/core/logging_helpers.py 분리)
def emit_llm_log(level: str, payload: dict) -> None:
    """LLM_LOG 표준 emitter — timestamp 자동 부착."""
    payload = {**payload, "timestamp": datetime.datetime.now(datetime.timezone.utc).isoformat()}
    line = f"[LLM_LOG] {json.dumps(payload, ensure_ascii=False)}"
    if level == "error":
        logger.error(line)
    else:
        logger.debug(line)
```

기존 5곳의 `logger.debug(f"[LLM_LOG] ...")` 를 모두 `emit_llm_log(...)` 호출로 치환한다.

### 2.3 latency 측정

`/chat` 의 `event_generator()` 시작 시각을 capture, 응답·에러 페이로드 직전에 차이 계산:

```python
import time
request_started_at = time.perf_counter()

# response 시:
latency_ms = int((time.perf_counter() - request_started_at) * 1000)
emit_llm_log("debug", {**response_log, "latency_ms": latency_ms})
```

`/api/rag/search` 도 동일 패턴을 적용.

### 2.4 토큰 / 모델 추출

langchain `astream` 의 마지막 chunk 가 `usage_metadata` 와 `response_metadata` 를 포함한다. 이를 누적해 응답 페이로드에 부착:

```python
final_chunk = None
async for chunk in llm.astream(messages):
    final_chunk = chunk
    ...

usage = getattr(final_chunk, "usage_metadata", None) or {}
model_name = getattr(final_chunk, "response_metadata", {}).get("model_name") or getattr(llm, "model_name", None)

response_log = {
    ...,
    "model": model_name,
    "input_tokens":  usage.get("input_tokens"),
    "output_tokens": usage.get("output_tokens"),
    "latency_ms": latency_ms,
}
```

> 모델/제공자에 따라 `usage_metadata` 가 `null` 일 수 있다. 그 경우 `null` 그대로 적재하여 Grafana 패널에서 결측치는 자연스럽게 무시되도록 한다.

### 2.5 RAG score 보강

`apps/api/main.py` 의 `rag_search_response` 페이로드에 다음 필드를 추가:

```python
scores = [r.get("score") for r in results if r.get("score") is not None]
top_score = max(scores) if scores else None
min_score = min(scores) if scores else None
```

text_matching 검색은 score 가 일괄 1.0 으로 채워지므로 패널에서 별도 분기 처리한다.

### 3.2 핵심 결정 사항
- **포트 미노출**: `loki`, `promtail`, `grafana` 모두 호스트 포트를 열지 않는다. 외부 접근은 nginx 의 `/grafana/` 만 허용.

- **`./loki_data`, `./grafana_data`**: 호스트 바인드 마운트로 데이터 영속화. **`.gitignore` 에 추가 필수**.
- **`-config.expand-env=true`**: Loki 설정파일 안에서 `${LOKI_RETENTION_PERIOD}` 치환 가능.

## 5. Promtail 설정 (`ops/promtail/promtail-config.yaml`)

```yaml
server:
  http_listen_port: 9080

clients:
  - url: http://loki:3100/loki/api/v1/push

positions:
  filename: /tmp/positions.yaml

scrape_configs:
  - job_name: docker
    docker_sd_configs:
      - host: unix:///var/run/docker.sock
        refresh_interval: 5s
    relabel_configs:
      # /<id>/<id>-json.log 의 컨테이너 이름을 라벨로
      - source_labels: ['__meta_docker_container_name']
        regex: '/(.*)'
        target_label: container
      - source_labels: ['__meta_docker_container_log_stream']
        target_label: stream

    pipeline_stages:
      # Docker JSON 로그 라인은 {"log":"...","stream":"stdout","time":"..."} 형태
      - json:
          expressions:
            output: log
            stream: stream
            time:   time
      - timestamp:
          source: time
          format: RFC3339Nano
      - output:
          source: output

      # llm-agent 컨테이너만 LLM_LOG 파싱
      - match:
          selector: '{container="llm-agent"}'
          stages:
            - regex:
                expression: '\[LLM_LOG\] (?P<llm_json>\{.*\})'
            - json:
                source: llm_json
                expressions:
                  type: type
                  model_type: model_type
                  request_id: request_id
                  thread_id: thread_id
                  user_id: user_id
                  model: model
            # 라벨 (저카디널리티)
            - labels:
                type:
                model_type:
            # 구조화 메타데이터 (고카디널리티)
            - structured_metadata:
                request_id:
                thread_id:
                user_id:
                model:
```

설계 포인트:
- `docker_sd_configs` 로 모든 컨테이너 자동 인식 → 사이드카 불필요.
- `match` 단계로 `llm-agent` 외 컨테이너는 LLM_LOG 파싱 없이 그대로 raw 적재.
- `structured_metadata` 단계는 Loki 3.x + `allow_structured_metadata: true` 와 함께 동작.

### 6.5 대시보드 LogQL 패널 예시

#### 운영 개요 — 시간대별 요청 수 (모델별)
```
sum by (model_type) (
  count_over_time({container="llm-agent", type="request"}[1m])
)
```

#### 운영 개요 — p95 응답 지연
```
quantile_over_time(0.95,
  {container="llm-agent", type="response"}
    | json
    | unwrap latency_ms
  [5m]
)
```

#### 에러율 (%)
```
( sum(count_over_time({container="llm-agent", type="error"}[5m]))
/ sum(count_over_time({container="llm-agent", type="request"}[5m])) ) * 100
```

#### 토큰 사용량 (input vs output, 모델별)
```
sum by (model) (
  rate({container="llm-agent", type="response"}
    | json
    | __error__ = ""
    | unwrap output_tokens [5m]
  )
)
```

#### 사용자 드릴다운 — 특정 user_id 의 trace
```
{container="llm-agent"}
  | json
  | user_id = `${user_id}`     # Grafana 변수
  | line_format "{{.type}}\t{{.request_id}}\t{{.full_response | default .message}}"
```

#### RAG 결과 0건 비율
```
( sum(count_over_time(
    {container="llm-agent", type="rag_search_response"}
      | json
      | results_count = "0"
    [5m]))
/ sum(count_over_time({container="llm-agent", type="rag_search_response"}[5m]))
) * 100
```

#### RAG 평균 top-1 score (벡터)
```
avg_over_time(
  {container="llm-agent", type="rag_search_response"}
    | json
    | search_method = "vector"
    | top_score != ""
    | unwrap top_score
  [5m]
)
```

각 패널은 dashboard JSON 에 `datasource: Loki` 로 사전 박제하여 첫 기동 시 즉시 노출.

## 7. nginx 변경 (`nginx.conf`)

기존 server block 안에 다음 location 추가:

```nginx
location /grafana/ {
    proxy_pass http://grafana:3000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # Grafana Live (websocket)
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

# Grafana Live SSE/WS 용 별도 엔드포인트 (선택)
location /grafana/api/live/ {
    proxy_pass http://grafana:3000/api/live/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
}
```

- `proxy_pass` 의 trailing slash 가 sub-path stripping 을 담당. Grafana 의 `serve_from_sub_path: true` 와 짝.
- 인증은 Grafana 자체 OIDC 가 처리하므로 nginx 는 단순 passthrough.

## 9. 구현 순서 (작업 단계) — 진행 상태 체크리스트

## 11.3 호스트 볼륨 권한 (운영 노트)

`loki_data/`, `grafana_data/` 는 호스트 바인드 마운트 방식이므로, 컨테이너의 비-root 사용자(loki=`10001`, grafana=`472`)가 쓰기 권한을 갖도록 첫 기동 전에 소유권을 보정해야 한다. 보정 없이 기동하면 두 컨테이너가 `permission denied` 로 무한 재시작한다.

```bash
# 호스트 sudo 가능 시
sudo chown -R 10001:10001 loki_data
sudo chown -R 472:472 grafana_data

# 또는 sudo 없이 docker 로 처리
docker run --rm -v "$PWD/loki_data:/data" alpine chown -R 10001:10001 /data
docker run --rm -v "$PWD/grafana_data:/data" alpine chown -R 472:472 /data
```

CI/배포 스크립트에서는 `docker compose up` 직전에 위 명령을 자동 실행하도록 한다.

## 13. 환경별 URL 설정 가이드 (Grafana ROOT_URL · nginx SNI · OIDC 콜백)

Grafana 는 자기 자산(/public/ 등) 과 리다이렉트 URL 을 **`GF_SERVER_ROOT_URL` 에 지정한 값** 으로 만들어 브라우저에 보낸다. 이 값이 사용자가 실제로 입력하는 URL 과 다르면 화면이 깨지거나 OIDC 로그인이 무한 루프에 빠진다. 따라서 환경이 바뀔 때마다 다음 4개 항목을 **함께** 정렬해 주어야 한다.

### 13.1 같이 움직여야 하는 4개 값

| 위치 | 값 | 비고 |
| :--- | :--- | :--- |
| `.env` `GRAFANA_ROOT_URL` | 사용자 브라우저가 실제로 보는 외부 URL | `/grafana` 까지 포함 |
| `.env` `OIDC_REDIRECT_URI` (Phase 06.1 시) | `${GRAFANA_ROOT_URL}/login/generic_oauth` | mwm-idp 의 콜백 URL 과 일치해야 함 |
| `nginx.conf` `server_name` | `GRAFANA_ROOT_URL` 의 호스트 부분 | TLS 인증서 SAN 과도 일치 |
| TLS 인증서 SAN | 위 호스트명 포함 | 자가 서명·공인 모두 동일 |

### 13.2 환경별 권장 값

| 환경 | `GRAFANA_ROOT_URL` | nginx `server_name` | TLS 인증서 |
| :--- | :--- | :--- | :--- |
| 개인 로컬 (host alias) | `https://llm-agent.mwm.local:21443/grafana` | `llm-agent.mwm.local` | `certs/cert.pem` (자가 서명, 현재 형상) |
| 노트북에서 빠른 확인만 | `https://localhost:21443/grafana` | `localhost` 추가 | 자가 서명 (cert SAN 에 localhost 포함 필요) |
| 사내 staging | `https://llm-stg.example.local/grafana` | `llm-stg.example.local` | 사내 CA |
| 운영 (production) | `https://llm.example.com/grafana` 또는 `https://grafana.example.com` | 운영 도메인 | Let's Encrypt 등 공인 CA |

> **주의**: `localhost` 와 `llm-agent.mwm.local` 을 둘 다 쓰고 싶다고 해서 `GRAFANA_ROOT_URL` 을 한쪽으로만 설정해 두면, 다른 쪽으로 접근한 사용자는 자산 로딩이 실패한다. 한 환경 = 한 ROOT_URL 원칙을 지킨다.

### 13.3 환경 전환 절차 (예: 로컬 → 운영)

1. **`.env` 갱신**
   ```bash
   GRAFANA_ROOT_URL=https://llm.example.com/grafana
   # Phase 06.1 활성 시
   GRAFANA_OIDC_AUTH_URL=https://idp.example.com/oauth/authorize
   GRAFANA_OIDC_TOKEN_URL=https://idp.example.com/oauth/token
   GRAFANA_OIDC_API_URL=https://idp.example.com/oauth/userinfo
   ```

2. **`nginx.conf` 의 `server_name` 변경**
   ```nginx
   server {
       listen 443 ssl;
       server_name llm.example.com;
       ssl_certificate     /etc/nginx/certs/llm.example.com.fullchain.pem;
       ssl_certificate_key /etc/nginx/certs/llm.example.com.key;
       ...
   }
   ```

3. **TLS 인증서 교체**: `certs/` 에 운영 도메인용 `cert.pem` / `key.pem` 또는 fullchain 적용. 공인 CA 권장.

4. **OIDC 클라이언트 콜백 URL 재등록 (Phase 06.1 적용 시)**
   - mwm-idp(또는 운영 IdP) 의 grafana 클라이언트에서 콜백 URL 을 새 값으로 변경:
     `https://llm.example.com/grafana/login/generic_oauth`

5. **재기동**
   ```bash
   docker compose up -d --force-recreate grafana nginx
   docker compose logs -f grafana | grep -i "HTTP Server Listen"
   ```
   `subUrl=/grafana` 가 출력되면 sub-path 모드 정상.

6. **검증**
   ```bash
   curl -k https://<ROOT_URL_HOST>:<PORT>/grafana/api/health
   # → {"database":"ok", ...}
   ```

### 13.4 운영 시 추가 권장사항

- **TLS 종료**: 운영에서는 nginx 가 공인 인증서로 TLS 종료, 내부 컨테이너 간 통신은 평문(HTTP). 본 형상과 동일.
- **Loki 보관 기간**: 운영 컴플라이언스에 따라 `LOKI_RETENTION_PERIOD` 를 조정 (예: 90일 = `2160h`).
- **디스크 모니터링**: `loki_data/` 는 트래픽에 따라 빠르게 커진다. 호스트에서 별도 감시.
- **`.env` 보호**: 운영 비밀번호/secret 은 컨테이너 호스트의 secrets manager(예: docker secrets, vault) 로 주입 권장.
- **OIDC 적용**: 운영 환경에서는 임시 admin 로그인을 비활성화하고 반드시 SSO(Phase 06.1) 만 허용.
