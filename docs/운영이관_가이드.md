# 운영이관 가이드 (production)

> 본 가이드는 현재 단일 호스트 `docker-compose.yml` (개발 환경) 을 production 으로
> 이관할 때의 변경 사항을 운영자가 그대로 따라 적용할 수 있도록 단계별로 기술한다.
> 적용 시점은 **Phase 07 (Tool Lab)** 까지가 prod 에 배포되는 시점이며, 이후 단계는
> 본 문서의 §10 체크리스트로 회귀 검증한 뒤 추가하면 된다.
>
> 동일 디렉터리의 `배포가이드.md` 가 "어떻게 빌드/실행" 의 단일 호스트 절차라면,
> 본 문서는 "어떤 컴포넌트를 prod 인프라로 옮길지 + 그 때 무엇이 달라져야 하는지"
> 의 분리·매핑 가이드이다.

---

## 1. 개요

### 1.1 핵심 변경점 (dev → prod 5가지)

| 항목 | dev (현재 compose) | prod (이관 후) |
|---|---|---|
| **관찰성 스택** | 같은 호스트에 `loki` + `promtail` + `grafana` 컨테이너 함께 기동 | **중앙 관찰성 인프라 (별도 호스트/클러스터)**. prod 호스트엔 promtail 한 개만 (혹은 fluent-bit/vector) 두고 중앙 Loki 로 push, Grafana 는 전사 공용 인스턴스 |
| **LLM 게이트웨이** | OpenAI hosted (`api.openai.com`) 또는 빈 BASE_URL | **사내 vLLM 클러스터** (gpt-oss, Qwen3-Coder 등). `*_BASE_URL` + `*_TOOL_PARSER` 명시 |
| **데이터 영속성** | `./qdrant_data`, `./postgres_data` 호스트 볼륨 | **외부 관리형 / 별도 클러스터** (RDS, Qdrant Cloud 또는 사내 클러스터) 권장. 호스트 볼륨도 가능하나 백업 정책 필수 |
| **secret 관리** | `.env` 평문, 레포 외부 | **Docker secrets / Vault / sops / Kubernetes Secret** 등 운영팀 표준에 따름. `.env` 는 컨테이너 기동 직전 마운트 |
| **TLS / 도메인** | self-signed + `*.mwm.local` host alias | **공식 도메인 + 사내 CA (또는 Let's Encrypt) 정식 인증서**, nginx (또는 외부 L7) 가 SSL termination |

### 1.2 본 가이드의 범위

- 다룬다: `.env` 분리, compose 분리, promtail/loki/grafana 분리, vLLM 모델별 환경변수 매트릭스, 시드/마이그레이션, 배포·검증·롤백
- 다루지 않는다: 사내 Kubernetes 표준, 사내 인증서 발급 절차, vLLM 자체 운영 (이건 모델팀 영역)

## 2. 아키텍처

### 2.1 dev — 단일 호스트

```

```

### 2.2 prod — 분리

```
┌────────────────────────────┐    ┌──────────────────────────────┐
│ prod host (app)            │    │ central observability        │
│                            │    │                              │
│  nginx (또는 외부 L7)      │    │  loki ←──┐                   │
│    ↓                       │    │  grafana │                   │
│  llm-agent  ─→ qdrant ✱   │    └─────────┬┴─────────────────-─┘
│    ↓        ─→ postgres ✱ │              │ TLS push
│  stdout                    │              │
│    ↓                       │              │
│  promtail (사이드카)─────────────────────-─┘
└──────────────┬─────────────┘
               │ OpenAI API

        ┌──────────────────────────┐
        │ vLLM cluster (GPU)       │

        └──────────────────────────┘

✱ qdrant / postgres 는 외부 관리형 서비스로 분리 권장
```

## 3. 사전 준비 (prerequisites)

### 3.1 인프라 체크리스트

| 항목 | 요구 |
|---|---|
| **prod 호스트** | Docker 24+, docker-compose v2, 최소 4 vCPU / 8 GB RAM (앱 단독), 디스크 50GB+ |
| **PostgreSQL** | 14+ (asyncpg 호환). UTF-8 collation. 권한: `CREATE TABLE`, `ALTER TABLE` (Phase 07 의 `ADD COLUMN IF NOT EXISTS` 가 시작 시 실행됨) |
| **Qdrant** | 1.10+ (Phase 02 의 dense vector + Phase 03 의 hybrid 가정) |
| **vLLM 노드** | OpenAI 호환 `/v1/chat/completions` + `/v1/models` 응답. Tool calling 필요 모델은 `--enable-auto-tool-choice --tool-call-parser <parser>` 옵션 켜져 있어야 함 |
| **중앙 Loki** | 2.9+ (혹은 Loki 3.x). `allow_structured_metadata: true` 권장 (현 promtail 설정이 structured_metadata 를 보냄) |
| **중앙 Grafana** | 10+ (derived field, Loki datasource OAuth 또는 service account) |
| **IDP** | OIDC. prod 용 클라이언트 ID/secret 발급. redirect_uri 등록 |

### 3.2 네트워크 / DNS / TLS

```
llm-agent.<prod-domain>          → prod host (nginx 443)
grafana.<prod-domain>            → 중앙 Grafana (분리)
idp.<prod-domain>                → 사내 IDP

prod-host  →  central-loki:3100  (LB 또는 직결, mTLS 권장)
prod-host  →  vllm-gptoss:8001   (사내망)
prod-host  →  vllm-qwen3:8002    (사내망)
prod-host  →  postgres:5432      (관리형)
prod-host  →  qdrant:6333        (관리형 또는 사내)
```

방화벽 / Security Group:
- 인바운드: 443 (nginx 또는 LB)
- 아웃바운드: Loki/vLLM/Postgres/Qdrant 포트 + IDP HTTPS

## 4. 환경 변수 (.env 분리)

### 4.1 디렉터리 구조 권장

```
repo/
  docker-compose.yml          # dev (그대로 유지)
  docker-compose.prod.yml     # prod (신규 추가; §5)
  sample.env                  # dev 템플릿 (그대로)
  deploy/
    envs/
      prod.env.template       # prod 템플릿 (커밋, 실제 값은 secret store)
    secrets/                  # gitignored. CI/CD 가 주입
      prod.env
```

`.gitignore` 에 다음 추가:
```
deploy/secrets/
```

### 4.3 Secret 관리

레포에 커밋되는 `deploy/envs/prod.env.template` 은 **키 이름만** 들어있는 빈 템플릿이다. 실제 값은 다음 중 한 가지 방식으로 주입:

| 방식 | 동작 |
|---|---|
| **Docker secret + entrypoint 합성** | 배포 직전 CI 가 `prod.env` 를 호스트의 `deploy/secrets/` 에 떨군다 (TTL 짧게, root:root 0600) |
| **HashiCorp Vault** | sidecar 또는 init-container 가 vault read 후 env 로 inject (Kubernetes 사용 시) |
| **sops + age** | 레포에 암호문으로 커밋, 배포 시점 복호화. CI runner 에 key 보관 |
| **운영팀 표준 Secret Manager** | AWS Secrets Manager / GCP Secret Manager 등 |

**금지 사항**:
- `.env` 평문을 git 에 커밋
- `docker inspect` 로 secret 이 그대로 노출되는 형태 (가능하면 env 대신 file mount + 앱에서 읽도록 변환)
- Grafana derived field URL 등에 secret 포함

## 5. docker-compose 분리

### 5.1 prod 용 compose 작성

`docker-compose.prod.yml` 을 신규 추가. 핵심 변경:
- **`loki`, `promtail`, `grafana` 서비스 통째로 제거**
- `nginx` 는 외부 L7 사용 시 제거, 호스트 nginx 가 SSL termination 하면 유지
- `db`, `qdrant` 는 외부 관리형으로 분리되면 제거
- `env_file` 을 `./deploy/secrets/prod.env` 로 변경
- 모든 이미지는 prod registry tag 사용 (`registry.example.com/llm-agent:<git-sha>`)

예시 (앱 단독 — Postgres/Qdrant 외부 관리형 가정):

```yaml
# docker-compose.prod.yml
services:
  llm-agent:
    image: registry.example.com/llm-agent:${LLM_AGENT_TAG:-latest}
    container_name: llm-agent
    ports:
      - "127.0.0.1:28000:8000"   # 외부 LB 가 가져가도록 loopback 만 노출
    env_file:
      - ./deploy/secrets/prod.env
    extra_hosts:
      - "idp.<prod-domain>:host-gateway"   # IDP 가 사내 DNS 미해결이면 사용
    restart: always
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:8000/api/config"]
      interval: 30s
      timeout: 5s
      retries: 5

  promtail:
    image: grafana/promtail:latest
    container_name: promtail
    command: -config.file=/etc/promtail/config.yaml
    volumes:
      - ./ops/promtail/promtail-prod.yaml:/etc/promtail/config.yaml:ro
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
      - /var/run/docker.sock:/var/run/docker.sock:ro
    depends_on:
      - llm-agent
    restart: always

networks:
  default:
    name: llm-network
```

### 5.2 운용 명령

```bash
# 빌드 — prod registry 로 push
docker build -t registry.example.com/llm-agent:$(git rev-parse --short HEAD) .
docker push registry.example.com/llm-agent:$(git rev-parse --short HEAD)

# prod 호스트에서
export LLM_AGENT_TAG=<git-sha>
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d
```

### 5.3 override 방식 (대안)

`docker-compose.yml` 을 그대로 두고 `docker-compose.override.prod.yml` 로 변경분만 기술 후

```bash
docker compose -f docker-compose.yml -f docker-compose.override.prod.yml up -d
```

— 가능하나 운영팀이 prod 의 최종 합성 결과를 한 화면에서 보기 어려워 **별도 파일 권장**.

## 6. 관찰성 외부화

### 6.1 옵션 A: prod 호스트의 promtail 만 두고 중앙 Loki 로 push (권장)

`ops/promtail/promtail-prod.yaml` 신규 작성. 현재 dev `promtail-config.yaml` (참고: `ops/promtail/promtail-config.yaml:5-6`) 의 `clients.url` 만 중앙 Loki 로 변경하면 된다. 인증을 더하는 경우 예:

```yaml
# ops/promtail/promtail-prod.yaml
server:
  http_listen_port: 9080
  grpc_listen_port: 0

clients:
  - url: https://loki.<central-domain>/loki/api/v1/push
    tenant_id: llm-prod                       # multi-tenant 운영 시
    basic_auth:
      username: ${LOKI_PUSH_USER}
      password: ${LOKI_PUSH_PASSWORD}
    # 또는 bearer_token / mTLS 사용
    tls_config:
      ca_file: /etc/ssl/certs/ca-bundle.crt   # 사내 CA 신뢰
    backoff_config:
      min_period: 500ms
      max_period: 5m
      max_retries: 10
    external_labels:
      env: prod
      service: llm-agent

positions:
  filename: /tmp/positions.yaml

scrape_configs:
  - job_name: llm-agent
    docker_sd_configs:
      - host: unix:///var/run/docker.sock
        refresh_interval: 5s
    relabel_configs:
      - source_labels: ['__meta_docker_container_name']
        regex: '/llm-agent'
        action: keep
      - source_labels: ['__meta_docker_container_name']
        regex: '/(.*)'
        target_label: container
      - source_labels: ['__meta_docker_container_log_stream']
        target_label: stream
    # pipeline_stages 는 dev 와 동일 — [LLM_LOG] 필터 + JSON 추출 + 라벨링
    pipeline_stages:
      - json:
          expressions: { output: log, stream: stream, time: time }
      - timestamp: { source: time, format: RFC3339Nano }
      - output: { source: output }
      - match:
          selector: '{container="llm-agent"} !~ "\\[LLM_LOG\\]"'
          action: drop
          drop_counter_reason: not_llm_log
      - regex:
          expression: '\[LLM_LOG\]\s+(?P<llm_json>\{.*\})'
      - json:
          source: llm_json
          expressions:
            type: type
            model_type: model_type
            request_id: request_id
            thread_id: thread_id
            user_id: user_id
            model: model
            search_method: search_method
      - output: { source: llm_json }
      - labels:
          type:
          model_type:
          search_method:
      - structured_metadata:
          request_id:
          thread_id:
          user_id:
          model:
```

**중요**: 위 `pipeline_stages` 는 dev 와 동일하게 유지해야 한다. 그래야 중앙 Loki 에 도착한 라인의 라벨/필드 구조가 똑같아지고, 우리가 만든 Grafana 대시보드 (`ops/grafana/dashboards/admin/*.json`, `user/*.json`) 가 그대로 동작한다.

### 6.2 옵션 B: 호스트의 fluent-bit / vector 가 수집

운영팀이 이미 호스트 표준 로그 에이전트를 운영 중이면 promtail 을 빼고 Docker JSON-file driver 의 stdout 만 흘려준다. 단, 위 `pipeline_stages` 의 LLM_LOG 추출 로직을 동등하게 fluent-bit/vector 측에 옮겨야 한다. 그렇지 않으면 `| json` 쿼리가 동작하지 않아 Grafana 대시보드가 빈다.

### 6.3 중앙 Grafana 에 대시보드 배포

`ops/grafana/dashboards/` 아래의 JSON 4개 (admin/operational-overview · rag-quality · toollab · user-drilldown, user/) 를 중앙 Grafana 로 옮긴다.

| 방법 | 절차 |
|---|---|
| **GitOps (권장)** | terraform-provider-grafana 또는 grafana-operator (Kubernetes) 로 레포의 JSON 을 declarative 하게 적용 |
| **수동 import** | Grafana UI → Dashboards → New → Import → JSON 붙여넣기. 폴더는 `LLM Admin` / `LLM User` 로 분리 (`ops/grafana/provisioning/dashboards/dashboards.yaml` 의 폴더명과 동일) |
| **API import** | `curl -X POST https://grafana.<central>/api/dashboards/db -H "Authorization: Bearer <service-account-token>" -d @ops/grafana/dashboards/admin/toollab.json` |

**Datasource 설정** (중앙 Grafana 에 이미 있을 가능성 높음):
- `uid: loki` 로 prod tenant 가리키는 Loki datasource 등록 — `ops/grafana/provisioning/datasources/loki.yaml:6` 참조

## 7. vLLM + 모델 매트릭스

### 7.1 모델별 tool_call_parser 매핑

`docs/P07_요구사항.md §4.3.1` 의 서빙 매트릭스를 운영자가 그대로 적용할 수 있도록 정리.

| 모델 family | `--tool-call-parser` | `*_TOOL_PARSER` env | reasoning 모델 여부 |
|---|---|---|---|
| **gpt-oss-** (120B/20B 등) | `openai` | `openai` | 둘 다 가능 |
| **Qwen3-Coder-** | `qwen3_coder` | `qwen3_coder` | chat 위주 |
| **Qwen3-Instruct / Qwen2.5-Instruct** | `hermes` | `hermes` | chat |
| **Llama-3.x-Instruct** | `llama3_json` | `llama3_json` | chat |
| **Mistral-Instruct** | `mistral` | `mistral` | chat |
| OpenAI hosted (api.openai.com) | — | 빈 값 | — |

> 본 앱은 parser 값을 **검증하지 않는다** — 그대로 LLM_LOG 라벨로 흘려서 Grafana 에서
> 모델 × parser 별 평균 iterations·latency 를 비교하는 데만 쓴다. 따라서 위 표는
> "vllm 가 실제로 그 parser 로 떠 있어야 한다" 는 운영 약속.

### 7.2 vLLM 서빙 명령 예시

```bash
# gpt-oss-120b (chat + tool calling)
vllm serve openai/gpt-oss-120b \
  --host 0.0.0.0 --port 8001 \
  --api-key "${VLLM_API_KEY}" \
  --enable-auto-tool-choice \
  --tool-call-parser openai

# Qwen3-Coder-30B (chat + tool calling)
vllm serve Qwen/Qwen3-Coder-30B-A3B \
  --host 0.0.0.0 --port 8002 \
  --api-key "${VLLM_API_KEY}" \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder
```

대응하는 prod `.env`:

```bash
CHAT_LLM_BASE_URL=http://vllm-gptoss:8001/v1
CHAT_LLM_MODEL=openai/gpt-oss-120b
CHAT_LLM_LABEL=GPT-OSS 120B
CHAT_LLM_API_KEY=${VLLM_API_KEY}
CHAT_LLM_USE_TEMPERATURE=true
CHAT_LLM_TOOL_PARSER=openai

REASONING_LLM_BASE_URL=http://vllm-qwen3:8002/v1
REASONING_LLM_MODEL=Qwen/Qwen3-Coder-30B-A3B
REASONING_LLM_LABEL=Qwen3-Coder 30B
REASONING_LLM_API_KEY=${VLLM_API_KEY}
REASONING_LLM_USE_TEMPERATURE=false
REASONING_LLM_TOOL_PARSER=qwen3_coder
```

### 7.3 chat vs reasoning 인스턴스 분리

같은 vLLM 프로세스로 두 모델을 동시에 띄울 수 없다 (한 프로세스 = 한 모델). prod 에선
**인스턴스 두 개** 운영하거나, 한 모델로 chat/reasoning 양쪽을 가리키는 단순화도 가능.
LangChain 게이트웨이는 `libs/core/llm.py` 의 `LLMGateway.get_chat_llm()` /
`get_reasoning_llm()` 로 모델 종류만 라우팅하므로 BASE_URL 만 잘 설정하면 된다.

### 7.4 임베딩 모델

`EMBEDDING_MODEL` 과 `EMBEDDING_DIM` 은 **반드시 일치**해야 한다. dim 이 안 맞으면
Qdrant 컬렉션 차원과 어긋나 search 가 죽거나 (Phase 02 부터) 첫 upsert 에서 실패.
다른 임베딩 모델로 갈아탈 때는 **Qdrant 컬렉션 재생성 + 재인덱싱** 필요.

### 7.5 헬스 체크

prod 호스트가 vLLM 노드를 끌어다 쓰려면 부트스트랩 시 `/v1/models` 가 200 을 반환해야
한다. 본 앱은 `libs/core/llm.py:get_llm_meta()` 가 매 호출 시점에 캐시 1분으로
`/v1/models` 의 첫 결과를 LLM_LOG 라벨로 가져간다. 죽어 있으면 `model_id` 라벨이
누락되어 Grafana 의 `model_id × parser` 매트릭스가 빈다.

## 8. 데이터 영속성 & 마이그레이션

### 8.1 PostgreSQL

- prod 에선 **관리형 PG 또는 별도 클러스터** 사용 권장. `docker-compose.prod.yml` 에서
  `db` 서비스를 제거하고 `DATABASE_URL` 만 외부를 가리키게 한다.
- **스키마 생성**: 앱이 처음 기동될 때 `apps/api/main.py:91-94` 의 `Base.metadata.create_all`
  이 모든 SQLAlchemy 모델로부터 테이블을 만든다. **운영 PG 사용자에 `CREATE TABLE` 권한
  필수**. 한 번 생성된 뒤엔 추가 마이그레이션 없음 (Phase 07 의 ADD COLUMN 은 별도).
- **Phase 07 idempotent 마이그레이션**: `libs/toollab/seed.py:bootstrap()` 이 매 startup 마다
  `ALTER TABLE tool_definitions ADD COLUMN IF NOT EXISTS is_public BOOLEAN NOT NULL DEFAULT FALSE` 를
  실행한다. **운영 PG 사용자에 `ALTER TABLE` 권한 필수**. 기존 row 에 영향 없음.
- **백업**: 표준 pg_dump 또는 관리형 PG 의 자동 스냅샷에 의존. `tool_definitions`,
  `tool_runs`, `chat_history`, `prompts`, `domains`, `collections` 가 주요 테이블.

### 8.2 Qdrant

- 외부 Qdrant 클러스터 (Qdrant Cloud 또는 사내 1~3 노드) 권장.
- `QDRANT_URL` + `QDRANT_API_KEY` 로 연결.
- **컬렉션은 앱 첫 사용 시 자동 생성**되며, `EMBEDDING_DIM` 변경 시 수동 삭제 + 재생성 필요.
- **백업**: Qdrant snapshot API (`POST /collections/<name>/snapshots`).

### 8.3 시드 도구

`libs/toollab/seed.py` 의 5개 (`list_users`, `get_user`, `create_user`, `update_user_email`, `onboard_new_team`) 가
startup 마다 idempotent 하게 등록된다. **prod 에서 데모용 시드를 끄려면**:
- 옵션 1: `TOOLLAB_ENABLED=false` — Tool Lab 자체 전면 off
- 옵션 2: `SEED_TOOLS` 리스트를 비운 별도 `seed.py` 를 마운트 (이미지 재빌드 필요)
- 옵션 3: 그대로 두고 운영 정책상 user 가 안 쓰도록 안내 (시드 도구는 메모리 only 라
  prod 데이터에 영향 없음)

### 9.3 롤백

```bash
export LLM_AGENT_TAG=<직전 git-sha>
docker compose -f docker-compose.prod.yml up -d llm-agent
```

**주의**: Phase 07 의 idempotent ALTER 는 `IF NOT EXISTS` 라 롤백해도 컬럼이 그대로
남는다 (해롭지 않음). 단, **이전 이미지가 `is_public` 컬럼을 모르는 버전이면** SQLAlchemy
ORM 이 unknown column 으로 죽지는 않지만 `INSERT` 시 default 값이 안 박혀 NULL constraint
violation 가능. 가능하면 1세대 이상 거꾸로 되돌리지 말 것.
