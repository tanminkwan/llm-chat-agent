## 2. 데이터 모델 설계

### 2.1 SQLAlchemy 모델 (`libs/core/models.py`)

```python
class Prompt(Base):
    __tablename__ = "prompts"

    id = Column(Integer, primary_key=True, index=True)
    user_id = Column(String(255), nullable=False, index=True)
    username = Column(String(255), nullable=True)
    title = Column(String(200), nullable=False)
    content = Column(Text, nullable=False)
    is_public = Column(Boolean, default=True, nullable=False)
    created_at = Column(DateTime(timezone=True), server_default=func.now(), nullable=False)
    updated_at = Column(DateTime(timezone=True),
                        server_default=func.now(),
                        onupdate=func.now(),
                        nullable=False)
```

- `user_id` 는 OIDC `sub` 를 그대로 저장. 향후 IDP 변경 시에도 안정적인 식별자 유지가 가능.
- `username` 은 표시 전용 컬럼. 등록 시점의 username 을 스냅샷으로 저장하여, 목록에서 추가 조회 없이 소유자명을 즉시 표시한다.
- `created_at / updated_at` 는 timezone-aware. `onupdate=func.now()` 로 수정 시점에 DB 가 자동 갱신.
- 테이블은 `startup` 핸들러의 `Base.metadata.create_all` 로 자동 생성된다.

### 2.2 인덱스 정책
- `user_id` 단일 인덱스: 본인 Prompt 조회가 가장 빈번하므로 필수.
- `is_public` 자체에는 별도 인덱스를 두지 않음. 데이터 카디널리티가 낮고, 조회 시 항상 `user_id` 기반 OR 절과 함께 사용되므로 user_id 인덱스만으로 충분.

## 3. 리포지토리 / 서비스 설계

### 3.1 PromptRepository (`libs/core/repository.py`)

```python
class PromptRepository(BaseRepository[Prompt]):
    async def search(self,
                     user_id: str,
                     include_others: bool = False,
                     title_keyword: Optional[str] = None) -> List[Prompt]:
        stmt = select(Prompt)
        if include_others:
            stmt = stmt.where(or_(Prompt.user_id == user_id,
                                  Prompt.is_public == True))
        else:
            stmt = stmt.where(Prompt.user_id == user_id)
        if title_keyword:
            stmt = stmt.where(Prompt.title.ilike(f"%{title_keyword}%"))
        stmt = stmt.order_by(Prompt.updated_at.desc())
        return (await self.db.execute(stmt)).scalars().all()
```

- 조회 규칙:
  1. `include_others=False` ⇒ 본인 것만.
  2. `include_others=True` ⇒ 본인 + `is_public=True` (타인의 비공개 Prompt 는 절대 노출 안 됨).
  3. 키워드는 `ilike` 로 대소문자 무시 부분일치.
- 정렬은 `updated_at DESC` 고정.

### 3.2 PromptService (`libs/core/service.py`)

| 메서드 | 권한 가드 | 예외 |
| :--- | :--- | :--- |
| `list_prompts(user_id, include_others, title_keyword)` | 인증된 User/Admin (라우트에서 체크) | – |
| `get_prompt(prompt_id, user_id)` | 본인 또는 `is_public=True` | 미존재: `ValueError`, 비공개 타인 건: `PermissionError` |
| `create_prompt(user_id, username, title, content, is_public)` | 인증된 User/Admin | – |
| `update_prompt(prompt_id, user_id, ...)` | **본인만** | 미존재: `ValueError`, 타인 건: `PermissionError` |
| `delete_prompt(prompt_id, user_id)` | **본인만** | 동일 |

라우트 계층은 `ValueError → 404`, `PermissionError → 403` 으로 매핑한다.

`💬 Chat 적용` 동작은 별도 서비스 메서드를 추가하지 않고 기존 `get_prompt` 를 재사용한다 — 동일한 권한 가드를 그대로 활용해 비공개 타인 건은 자연스럽게 차단된다.

### 4.3 응답 변환 헬퍼

```python
def _to_prompt_read(prompt, current_user_id: str) -> dict:
    return {
        "id": prompt.id,
        "user_id": prompt.user_id,
        "username": prompt.username,
        "title": prompt.title,
        "content": prompt.content,
        "is_public": prompt.is_public,
        "is_owner": prompt.user_id == current_user_id,
        "created_at": prompt.created_at,
```

### 5.3 Chat 헤더 변경


### 5.4 prompt.js 책임 분리

| 함수 | 책임 |
| :--- | :--- |
| `initPrompts()` | 진입 1회 초기화 — 현재 사용자 캐시, Enter 키 바인딩, 첫 검색. |
| `searchPrompts()` | 입력값 → `/api/prompts` 호출 → 테이블 렌더링. |
| `openPromptSidebar(mode, id)` | 사이드바 오픈 (`create / edit / show`). edit·show 는 `/api/prompts/{id}` 로 단건 fetch 후 폼 채움. |
| `closePromptSidebar()` | 오버레이/사이드바 비활성화. |
| `savePrompt(event)` | 모드별 POST/PUT 분기, 성공 시 사이드바 닫고 재검색. |
| `deletePrompt(id)` | confirm 후 DELETE → 재검색. |
| **`applyPromptToChat(id)`** | 단건 fetch → `#system-prompt-input.value` 채움 + `display:'block'` → `showView('chat', { push: true })`. |
| `_to_prompt_read` 결과의 `is_owner` 활용 | 액션 버튼(View / Edit / Delete) 노출 분기. `💬 적용` 은 모든 행에 공통 노출. |

#### 5.4.1 `applyPromptToChat(id)` 상세

```javascript
async function applyPromptToChat(promptId) {
    const res = await fetch(`/api/prompts/${promptId}`);
    if (!res.ok) {
        // 비공개 타인 건 등 권한 실패 시 detail 메시지 표시
        let detail = 'Prompt 를 불러올 수 없습니다.';
        try { const err = await res.json(); if (err.detail) detail = err.detail; } catch (_) {}
        alert(detail);
        return;
    }
    const item = await res.json();
    const sysInput = document.getElementById('system-prompt-input');
    if (!sysInput) { alert('Chat 화면이 준비되지 않았습니다.'); return; }
    sysInput.value = item.content || '';
    sysInput.style.display = 'block';   // 접힌 상태였다면 펼쳐서 사용자가 즉시 확인 가능
    showView('chat', { push: true });
}
```

- **단건 조회 재사용**: 별도 적용 API 를 추가하지 않고 표준 `GET /api/prompts/{id}` 를 사용해 권한 가드를 일관되게 적용.
- **DOM 직접 조작**: SPA 구조 덕분에 Chat 뷰의 `#system-prompt-input` 이 항상 DOM 에 상주하므로 비활성 뷰의 input 에도 직접 값을 쓸 수 있다.
- **자동 펼침**: 적용된 본문이 접힌 토글 안에 가려지지 않도록 `style.display='block'` 을 명시.
- **SPA 이동**: `showView('chat', { push: true })` 로 Chat 활성화 + 브라우저 히스토리에 push → 뒤로가기로 목록 복귀 가능.

## 6. 데이터 흐름 예시

### 6.1 신규 등록

```
1) 사용자가 /prompts 진입
     └─ app.js → showView('prompts') → initPrompts() → searchPrompts()
2) "+ 신규 Prompt 등록" 클릭 → openPromptSidebar('create')
3) 제목/내용 입력 + (옵션) 공유 체크해제 → savePrompt(event)
4) POST /api/prompts {title, content, is_public}
     ├─ FastAPI: 권한 검증 → PromptService.create_prompt
     ├─ DB INSERT (user_id = sub, username = preferred_username, server_default created_at/updated_at)
     └─ PromptRead 반환 (is_owner=true)
5) 사이드바 닫기 + searchPrompts() 재실행 → 목록 갱신
```

### 6.2 Chat 에 적용

```
1) /prompts 목록의 어느 행에서 [💬 적용] 클릭
2) GET /api/prompts/{id}
     ├─ 본인 또는 is_public=true → 200 + content 반환
     └─ 비공개 타인 건 → 403
3) #system-prompt-input.value = content; display = 'block'
4) showView('chat', { push: true })
     ├─ #view-chat .active 토글
     ├─ history.pushState({}, '', '/')
     └─ Chat 뷰 활성화 → 사용자는 즉시 메시지 입력 가능
5) 브라우저 ◀ → popstate → 다시 /prompts 활성화
```

## 7. 예외 / 보안 고려

| 시나리오 | 동작 |
| :--- | :--- |
| 비-User 그룹의 API 호출 | `403 권한이 없습니다.` |
| 존재하지 않는 `id` 조회/수정/삭제 | `404 Prompt not found` |
| 비공개 타 user Prompt 조회 시도 | `403 조회 권한이 없습니다.` |
| 타 user Prompt 수정/삭제 시도 | `403 수정 권한이 없습니다.` / `403 삭제 권한이 없습니다.` |
| **`💬 적용` 시 비공개 타인 건** | `GET /api/prompts/{id}` 가 403 → 클라이언트 alert 후 화면 이동 없이 중단 |
| `title` 의 SQL/HTML 인젝션 | SQL 은 SQLAlchemy 파라미터 바인딩으로 차단. 프론트는 `_escapeHtml` 로 출력 시 이스케이프. |
| 매우 긴 본문 | textarea 20행 + `resize: vertical` 로 가독성 확보. DB 는 `Text` 타입으로 길이 제약 없음. |
| 캐시된 옛 정적 자원 | `?v=11` 으로 일괄 갱신. |

## 8. 영향 범위 / 회귀 위험
- **백엔드**: 기존 RAG / Chat / Auth 엔드포인트에는 변경 없음. 신규 라우트만 추가.
- **DB**: `prompts` 테이블 신규 생성. 기존 `collections / domains` 와 무관.
- **프론트**: SPA 라우터 `VIEW_IDS` 에 `prompts` 가 추가됨. 기타 뷰의 활성화 로직은 그대로 유지.
- **CSS**: RAG 사이드바 셀렉터에 `.action-sidebar-panel` 가 OR 로 추가됨. 기존 `#action-sidebar` 동작은 동일.
- **`💬 적용` 의 부수 효과**: Chat 뷰의 `#system-prompt-input.value` 가 외부에서 수정될 수 있으나, 사용자가 명시적으로 트리거하므로 의도된 동작이며 채팅 메시지 송신 로직에는 영향 없음.

## 9. 테스트 계획

| 항목 | 검증 방법 |
| :--- | :--- |
| `prompts` 테이블 자동 생성 | 컨테이너 재기동 후 `\d prompts` 또는 INSERT 시 정상 동작 |
| User 권한 CRUD | User 계정으로 POST/GET/PUT/DELETE 정상 응답 |
| 비공개 격리 | A 사용자가 B 의 비공개 Prompt 를 `include_others=true` 로도 조회 불가 |
| 공개 공유 조회 | A 가 B 의 `is_public=true` Prompt 를 조회 가능, View 만 노출 |
| 타인 건 수정/삭제 차단 | 403 응답 |
| 제목 부분검색 | ILIKE `%키워드%` 가 케이스 무시로 동작 |
| Chat → Prompt 이동 | 시스템 프롬프트 영역의 `📝 Prompt 조회` 클릭 시 페이지 새로고침 없이 `/prompts` 활성화 |
| **`💬 적용` 정상 동작** | 본인/공개 타인 건에서 클릭 시 시스템 프롬프트 입력창에 본문이 채워지고 Chat 활성화, textarea 가 펼쳐진 상태 |
| **`💬 적용` 권한 차단** | 비공개 타인 건에 대해서는 적용 불가 (403 alert) |
| 긴 Prompt 가독성 | 수천 자 본문이 textarea 안에서 스크롤 가능 |
| 캐시 갱신 | `?v=11` 으로 신규 정적 자원 다운로드 |
| SPA 회귀 | Chat / RAG / Bulk / Admin 의 기존 시나리오 정상 동작 |
