# Phase 06 요구사항 정의서 (LLM 운영 로그 수집·시각화)

## 1. 개요
현재 `llm-agent` 컨테이너는 사용자/LLM 상호작용·RAG 검색에 대해 `[LLM_LOG]` 접두사를 가진 JSON 라인을 stdout 으로 흘리고 있다. 이 로그는 Docker 가 캡처해 저장하고 있을 뿐, **검색·집계·시각화 수단이 없어서 운영 중인 모델 사용 패턴, 응답 지연, 에러 흐름을 파악하기 어렵다**. 본 단계에서는 **Promtail → Loki → Grafana** 스택을 도입해 이 로그를 자동 수집·인덱싱·시각화하는 운영 가시성(Observability) 기반을 구축한다.

## 2. 목표
- **로그 자동 수집**: `llm-agent` 컨테이너의 stdout 을 Promtail 이 자동으로 tail 하여 Loki 에 적재.
- **구조화 적재**: `[LLM_LOG]` 라인의 JSON 본문을 자동 파싱하여 LogQL 로 즉시 필터링·집계 가능한 형태로 인덱싱.
- **운영 대시보드**: Grafana 에 사전 프로비저닝된 대시보드 3종(운영 개요 / 사용자 추적 / RAG 품질) 을 첫 기동 시 즉시 사용 가능하도록 제공.
- **단일 진입점·단일 인증**: 기존 nginx 리버스 프록시(`https://localhost:21443`) 뒤에 Grafana 를 두고, 기존 OIDC 와 동일한 IdP(`mwm-idp`) 를 통해 SSO 로 접근.
- **유연한 보관 기간**: Loki 보관 기간을 `.env` 환경변수로 조절할 수 있도록 한다.

## 3. 시스템 구성

### 3.1 추가되는 컨테이너
docker-compose 에 다음 3개의 서비스를 신규 추가하고 기존 `llm-network` 에 합류시킨다.

| 서비스 | 이미지 | 역할 |
| :--- | :--- | :--- |
| `loki` | `grafana/loki:latest` | 로그 저장소(단일 노드, filesystem 백엔드) |
| `promtail` | `grafana/promtail:latest` | Docker SD 로 컨테이너 로그를 tail → Loki 로 push |
| `grafana` | `grafana/grafana:latest` | 시각화 대시보드 + OIDC 인증 |

기존 `llm-agent`, `nginx`, `qdrant`, `db` 의 정의는 변경하지 않는다 (단, nginx 설정에 Grafana 라우트만 추가).

### 3.2 데이터 흐름
```
llm-agent (stdout JSON) ── Docker JSON log file ──► Promtail ── push ──► Loki ── LogQL ──► Grafana ◄── OIDC ── mwm-idp
                                                                                  ▲
                                                                                  │
                                                       사용자(브라우저) ─── https://localhost:21443/grafana ─── nginx
```

## 4. 로그 분석 결과 및 보강 요건

### 4.1 현재 적재 대상 로그 (그대로 유지, 마스킹 없음)
요구에 따라 **로그 본문은 가공·마스킹·축약 없이 전문 그대로 적재** 한다.

| `type` | 발생 지점 | 주요 필드 (현재) |
| :--- | :--- | :--- |
| `request` | `/chat` 진입 | `request_id`, `user_id`, `thread_id`, `model_type`, `messages[]` (system + history + 사용자 메시지) |
| `response` | `/chat` 스트림 종료 | `request_id`, `user_id`, `thread_id`, `full_response` |
| `error` | LLM 스트림 예외 | `request_id`, `user_id`, `thread_id`, `error` |
| `rag_search_request` | `/api/rag/search` 진입 | `request_id`, `user_id`, `collection_id`, `domain_id`, `query`, `search_method` |
| `rag_search_response` | RAG 결과 직후 | `request_id`, `user_id`, `results_count`, `results_metadata[]` |

### 4.2 추가로 적재해야 하는 필드 (애플리케이션 보강)
대시보드의 핵심 패널(응답시간, 토큰 사용량, RAG 품질 추이) 작성을 위해 다음 필드를 신규 추가한다.

| 필드 | 적용 대상 | 설명 |
| :--- | :--- | :--- |
| `timestamp` (ISO 8601, UTC) | 모든 `[LLM_LOG]` 라인 | LogQL 에서 라인 prefix 파싱 없이 직접 사용 |
| `latency_ms` | `response`, `error`, `rag_search_response` | request 시각 대비 종료 시각 차이 (ms) |
| `model` | `response` | 실제 LLM 모델 ID (`model_type` 외에 구체 모델명) |
| `input_tokens`, `output_tokens` | `response` | LLM `usage_metadata` 에서 추출 — 모델별 사용량/비용 분석용. 사용량 정보를 제공하지 않는 모델인 경우 `null` 허용 |
| `top_score`, `min_score` | `rag_search_response` | 검색 결과 품질 추이 패널 (벡터 검색에 한함) |

### 4.3 적재되지 않아야 하는 정보
- 비밀번호, OIDC access/id token, JWKS 서명값 등 인증 비밀정보는 어떤 로그에도 포함되지 않는다 (현재 코드도 미포함, 회귀 검증).
- 별도의 PII 마스킹은 수행하지 않는다 (요구 결정사항). 대신 `§7 보안` 의 OIDC 가드로 접근을 통제한다.

## 5. Promtail 파이프라인 요구사항

### 5.1 입력
- Docker SD(`docker_sd_configs`) 를 사용해 `llm-network` 의 모든 컨테이너 로그를 자동 인식하되, `container=llm-agent` 만 LLM_LOG 파이프라인 적용.

### 5.2 파싱
- 정규식으로 `[LLM_LOG] {json}` 부분만 추출 후 `json` stage 로 구조화.
- `[LLM_LOG]` 가 아닌 일반 INFO/DEBUG 로그(예: uvicorn access log)도 기본 라벨링되어 Loki 에 적재되지만, 별도 파싱은 하지 않는다.

### 5.3 라벨 정책 (카디널리티 관리)
- **라벨로 사용**: `container`, `service`, `type`, `model_type`
- **structured metadata 로 사용**: `request_id`, `thread_id`, `user_id`, `model`
- 카디널리티가 매우 높은 식별자(`request_id` 등) 를 라벨로 두면 인덱스 폭발이 발생하므로, Loki 3.x 의 structured metadata 또는 LogQL `| json` 단계에서 풀어쓰는 방식을 사용한다.

## 6. Loki 요구사항
- **단일 노드 + filesystem 백엔드**: Phase 06 단계의 로그량을 고려할 때 microservice/object storage 분리는 불필요.
- **데이터 디렉터리**: 호스트의 `./loki_data` 를 컨테이너에 바인드 마운트. `.gitignore` 에 추가하여 git 추적 제외.
- **보관 기간(retention)**: `.env` 의 환경변수 `LOKI_RETENTION_PERIOD` (예: `720h` = 30일) 로 제어.
  - Loki `limits_config.retention_period` 와 `compactor.retention_enabled: true` 에 동일 값을 주입.
  - 미설정 시 기본값 `720h` (30일).
- **압축 / 청크 세팅**: 별도 튜닝 없이 Loki 기본값 사용.

## 7. Grafana 요구사항

### 7.1 인증 — 2단계 적용 (현재 Phase 는 임시 admin, OIDC 는 후속 작업)

본 Phase 06 의 작업 범위 내에서는 **Grafana 인증을 OIDC 가 아닌 기본 admin 로컬 로그인** 으로 운영한다. OIDC 클라이언트 등록은 추후 사용자가 직접 mwm-idp 에 수행할 예정이며, 그 시점에 환경변수만 채우면 SSO 가 활성화되도록 설정 *틀(skeleton)* 만 미리 준비한다.

#### 7.1.A 현재 Phase 06 (구현 범위)
- `GF_AUTH_BASIC_ENABLED=true`.
- `GF_SECURITY_ADMIN_USER=admin`, `GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_ADMIN_PASSWORD}` (기본 비밀번호는 `.env` 에서 강제 설정).
- 익명 접근(`GF_AUTH_ANONYMOUS_ENABLED`) 은 비활성. 회원가입(`GF_USERS_ALLOW_SIGN_UP`) 도 비활성.
- nginx sub-path 프록시는 본 Phase 에 포함되어 외부에서는 `https://localhost:21443/grafana` 로만 접근 가능.

#### 7.1.B 후속 작업 (Phase 06.1 — OIDC SSO)
- 기존 `mwm-idp` 를 IdP 로 하는 Generic OAuth 연동.
- 신규 클라이언트(`grafana-client`) 등록 + 콜백 URL: `https://localhost:21443/grafana/login/generic_oauth`.
- 그룹 매핑:
  | OIDC `groups` | Grafana Role | 비고 |
  | :--- | :--- | :--- |
  | `Admin` | Admin | 대시보드 편집/생성/삭제, 데이터소스 관리 |
  | `User` | Viewer | 대시보드 조회만 가능 (편집 불가) |
  | 그 외 | 접근 불가 | OIDC `role_attribute_strict: true` 로 차단 |

  > **로그 내용은 대화 본문 전체를 포함하므로 일반 User 의 조회 허용 자체에도 노이즈/리스크가 있다.** SSO 일관성을 위해 Viewer 로 허용하되, 운영 정책상 더 강하게 막을 필요가 생기면 `Admin only` 로 손쉽게 좁힐 수 있도록 매핑은 환경변수로 제어한다 (`GRAFANA_ALLOWED_GROUPS`, `GRAFANA_ADMIN_GROUPS`).

- Phase 06.1 에서는 `GF_AUTH_BASIC_ENABLED=false`, `GF_AUTH_DISABLE_LOGIN_FORM=true` 로 전환하고 SSO 만 허용한다.
- Phase 06 의 `sample.env` 에 OIDC 변수 자리(주석 처리) 를 미리 마련해 두어, Phase 06.1 시 활성화 시점을 단순화한다.

### 7.2 외부 노출 — nginx 리버스 프록시
- 외부 접근은 **`https://localhost:21443/grafana`** 단일 경로만 허용.
- nginx 에 `location /grafana/ { proxy_pass http://grafana:3000/; }` 라우트 추가 (websocket 헤더 포함).
- Grafana 컨테이너 자체 포트는 호스트로 노출하지 않는다 (`ports` 미설정 또는 내부 네트워크만).
- Grafana 설정의 `server.root_url` 와 `server.serve_from_sub_path` 를 sub-path 모드에 맞게 설정.

### 7.3 데이터소스
- `loki` 를 기본 데이터소스로 사전 프로비저닝(`provisioning/datasources/*.yaml`).
- 테스트 시 `Save & Test` 가 자동으로 통과하는 상태로 출고.

### 7.4 대시보드 (사전 프로비저닝)
세 종을 `provisioning/dashboards/*.json` 으로 동봉, 첫 기동 시 즉시 노출.

#### 7.4.1 LLM 운영 개요 (Operational Overview)
- 시간대별 요청 수 — `model_type` 별 stack
- 응답 지연 분포 (p50 / p95 / p99) — `latency_ms`
- 에러율 (% over time) — `count(type="error") / count(type="request")`
- 동시 사용자 수 추이 — distinct `user_id`
- 토큰 사용량 (input vs output) — 모델별 누적

#### 7.4.2 사용자/스레드 추적 (User Drilldown)
- 사용자별 호출 빈도 Top 20
- `user_id` 입력 필터 → 본인의 모든 thread / request / response 타임라인
- `request_id` 입력 필터 → request → (RAG 호출들) → response/error 의 단일 trace 뷰

#### 7.4.3 RAG 품질
- 알고리즘별(vector vs text_matching) 호출 비율
- 결과 0건 비율 (poor recall 지표)
- 평균 top-1 score 추이 (벡터 검색에 한함)
- 컬렉션·도메인별 질의 분포 heatmap

### 7.5 알림 (제외)
- Grafana Alerting 은 **본 Phase 에서 제공하지 않음**.
- Alerting UI 자체는 비활성화 또는 비노출 (대시보드 메뉴에서 제거).
- 향후 단계에서 추가할 여지는 남기되 본 단계 DoD 에는 포함하지 않는다.

## 8. nginx 변경사항
- `nginx.conf` 에 다음을 추가:
  ```nginx
  location /grafana/ {
      proxy_pass http://grafana:3000/;
      proxy_set_header Host $host;
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For  $proxy_add_x_forwarded_for;
      proxy_set_header X-Forwarded-Proto $scheme;

      # Grafana live (websocket)
      proxy_http_version 1.1;
      proxy_set_header Upgrade $http_upgrade;
      proxy_set_header Connection "upgrade";
  }
  ```
- 기존 `llm-agent` 라우트(`/`, `/api`, `/auth`, `/static` 등) 는 변경하지 않는다.

## 10. 호환성 / 비기능 요구사항
- **백엔드 API 시그니처 변경 금지**: `/chat`, `/api/*` 엔드포인트의 입출력 형식은 그대로. 4.2 의 보강 필드는 **로그에만** 추가되며 클라이언트 응답 페이로드에는 포함하지 않는다.
- **회귀 영향 없음**: 신규 컨테이너 3종은 기존 서비스에 무관하게 부수적으로 동작. 어떤 이유로든 Loki/Promtail/Grafana 가 다운되어도 `llm-agent` 의 동작에는 영향이 없어야 한다 (로그 stdout 은 항상 가능).
- **테스트 전략**: 로그 보강 부분은 단위 테스트로 검증. Loki/Grafana 자체는 외부 OSS 컴포넌트이므로 컨테이너 기동·헬스체크 수준의 통합 검증으로 충분.
