# Phase 4 요구사항 정의서 (RAG 통합 관리 콘솔)

## 1. 개요
기존 모달 방식의 UI를 탈피하여 일반 사용자(User)도 접근 가능한 전체 화면 기반의 RAG 관리 및 지식 검색 시스템을 구축한다.

## 2. 주요 요구사항

### 2.1 UI/UX 전환
- **전체 화면 구성**: `/static/rag.html` 경로의 독립된 페이지에서 RAG 데이터 관리.
- **사용자 권한 확대**: Admin뿐만 아니라 일반 **User**도 접근 및 기능 사용 가능.
- **사이드바 통합**: 채팅 화면과 지식 관리 화면을 자유롭게 이동할 수 있는 내비게이션 제공.

### 2.2 RAG 데이터 체계
- **Collection (RAG 유형)**: 데이터의 구조적 유형 정의.
    - **식별자**: 실제 Qdrant의 **콜렉션 명(Table Name)**을 식별자로 사용한다.
    - **제약**: 중복 등록이 불가능하며, 한 번 등록된 콜렉션 명은 **수정할 수 없다**.
- **Domain (지식 분야)**: 데이터의 지식 카테고리 정의 (예: 의학, 경제, 일상 등).
- **데이터 관계**:
    - 하나의 Collection(유형) 내에 여러 Domain(분야)의 데이터가 존재할 수 있음.
    - Qdrant의 각 포인트(문서 조각) 페이로드에 `domain_id`를 필수로 포함.


### 2.3 검색 및 필터링 기능
- **리스트 조회 조건**:
    - **Collection 선택**: 검색 대상 유형 결정 (필수).
    - **Domain 선택**: 특정 분야로 필터링 (선택).
    - **입력 텍스트**: 실제 지식 검색을 위한 쿼리 문자열.
- **검색 방식**: 각 Collection에 정의된 `search_method`(Vector, BM25 등)를 동적으로 적용.

### 2.4 성능 및 관리
- **Qdrant 인덱스**: 각 콜렉션의 `domain_id` 페이로드 필드에 대해 **Payload Index**를 생성하여 필터링 속도 최적화.
- **CRUD**: 유형(Collection) 및 분야(Domain)에 대한 전체 화면 기반의 등록/수정/삭제 기능.
